## Обзор чат-системы

Документ описывает архитектуру и ключевые компоненты чат-системы Knowledge Base. Чат — это AI-ассистент, который отвечает на вопросы пользователя, используя инструменты (tools) для доступа к базе знаний и Git-репозиторию.

**Связанные документы:**
- [API Reference](api-reference.md) — эндпоинты `ChatController`
- [AI-инструменты](ai-инструменты.md) — полный список инструментов
- [Модели данных](модели-данных/.content.md) — сущности `ChatTopicEntity`, `ChatMessageEntity`
- [Конфигурация](конфигурация.md) — `ChatConfig`, параметры AI

---

### 1. Архитектура чата

```
Пользователь (Frontend)
    │  SSE / HTTP
    ▼
ChatController  (/api/chat)
    │
    ├── ChatClient (Spring AI)
    │       ├── ChatModel (OpenAI-совместимый)
    │       ├── ChatMemory → ChatMemoryService → PostgreSQL (chat_message)
    │       ├── ToolCallbacks (5 функций) → RecordingToolCallback → ToolInvocationCollector
    │       └── System Prompt (@Value classpath:prompt/sys.md)
    │
    ├── ChatTopicRepository → PostgreSQL (chat_topic)
    ├── SummarizeService (фоновая суммаризация)
    └── JiraChatService (чат по задаче Jira)
```

**Ключевые особенности:**
- **Стриминг через SSE** — ответы передаются в реальном времени через `SseEmitter`
- **Инструменты (tools)** — AI может вызывать функции для поиска документов, чтения файлов, истории коммитов
- **Компактные результаты** — `CompactToolResultConverter` сжимает результаты в `resultGist` для отображения в UI
- **Сохранение tool calls** — `ChatMemoryService.saveToolCalls()` сохраняет SYSTEM-сообщение с JSON tool calls (кроме скрытых инструментов)
- **Частичное сохранение при обрыве SSE** — при разрыве соединения накопленный ответ сохраняется как `AssistantMessage`
- **Суммаризация** — длинные диалоги автоматически сжимаются, чтобы не превысить контекстное окно
- **Jira-чаты** — можно создать чат, предзагруженный контентом задачи Jira и страницы Confluence

### 2. ChatController — точка входа

`ChatController` (`/api/chat`) — единственный контроллер чата. Зависимости:

| Зависимость | Назначение |
|---|---|
| `ChatClient` | Основной клиент Spring AI для общения с моделью |
| `ChatMemory` | Управление историей сообщений (скользящее окно) |
| `ChatTopicRepository` | CRUD для топиков чата |
| `ChatMemoryService` | Персистентное хранение сообщений в БД |
| `SummarizeService` | Фоновая суммаризация длинных диалогов |
| `JiraChatService` | Создание чатов с контекстом Jira |

**9 эндпоинтов** (подробнее в [API Reference](api-reference.md)):
- `POST /api/chat/stream` — стриминг ответа AI
- `POST /api/chat/streamTest` — тестовый стриминг (демонстрация tool calls)
- `POST /api/chat` — синхронный запрос
- `GET /api/chat` — список чатов пользователя
- `GET /api/chat/chat` — конкретный чат с сообщениями
- `DELETE /api/chat/chat` — удаление чата
- `POST /api/chat/topic` — установка/обновление топика
- `POST /api/chat/jira` — создание Jira-чата
- `POST /api/chat/jira/{id}/refresh` — обновление Jira-чата

При загрузке истории (`GET /api/chat/chat`) фильтруются сообщения типа `SYSTEM` — они не показываются пользователю.

#### Частичное сохранение при обрыве SSE

При разрыве соединения (клиент закрыл вкладку, сетевой сбой) `ChatController` сохраняет накопленный ответ ассистента:

- **`StringBuffer buffer`** — накапливает чанки текста ответа ассистента
- **`AtomicBoolean persisted`** — защита от двойного сохранения
- **`.doFinally(signal)`** — при `CANCEL` или `ON_ERROR` вызывает `persistPartial()`
- **`persistPartial()`** — сохраняет накопленный текст как `AssistantMessage` + tool calls через `chatMemoryService.saveToolCalls()`
- **Сброс буфера:** при `finishReason != null` (конец сегмента) буфер очищается (`buffer.setLength(0)`) — каждый сегмент сохраняется отдельно
- **`emitter.onCompletion()`** вызывает `dispose()` — разрыв клиентом → `CANCEL` → частичное сохранение

### 3. ChatClient и ChatConfig

`ChatConfig` — центральная конфигурация чата. Создаёт два бина:

#### 3.1 ChatMemory
```java
MessageWindowChatMemory.builder()
    .chatMemoryRepository(chatMemoryService)  // персистентное хранение
    .maxMessages(50)                          // скользящее окно
    .build();
```
- Хранит последние 50 сообщений в оперативной памяти
- При превышении — старые сообщения вытесняются (но сохраняются в БД через `ChatMemoryService`)

#### 3.2 ChatClient
- **ChatModel** — OpenAI-совместимая модель (через `OpenAiChatModel`)
- **System Prompt** — загружается через `@Value("classpath:prompt/sys.md") Resource sysPrompt` и передаётся в `.defaultSystem(sysPrompt)`
- **MessageChatMemoryAdvisor** — автоматически добавляет историю в каждый запрос
- **ToolCallingManager** — `ToolCallingManager.builder().toolCallbacks(...)` с 5 ToolCallbacks, обёрнутых в `RecordingToolCallback`:
  - `TopicFunction` — авто-именование чата
  - `MessageLookupFunction` — поиск сообщений по позиции
  - `DocumentFunction` — работа с документами базы знаний
  - `GitFunction` — работа с Git-репозиторием
  - `AttachmentFunction` — работа с вложениями

### 4. ChatMemoryService — персистентность сообщений

`ChatMemoryService` реализует `ChatMemoryRepository` и отвечает за сохранение сообщений в PostgreSQL.

**Таблица `chat_message`:**
| Поле | Тип | Описание |
|---|---|---|
| `id` | `long` | Первичный ключ |
| `conversation_id` | `String` | ID беседы |
| `content` | `String` | Текст сообщения |
| `type` | `MessageType` | USER, ASSISTANT, SYSTEM, TOOL |
| `position` | `long` | Порядковый номер в беседе |
| `summarized` | `boolean` | Суммаризировано ли |
| `summary` | `boolean` | Является ли само сообщение суммаризацией |
| `created_at` | `LocalDateTime` | Время создания |

**Ключевые методы:**
- `saveAll` — сохраняет новые сообщения, фильтруя уже существующие (по `IMessage`)
- `findByConversationId` — возвращает несуммаризированные сообщения
- `findChatMessageByConversationId` — возвращает не-summary сообщения (для отображения пользователю), фильтрует `MessageType.SYSTEM`
- **`saveToolCalls(conversationId, toolCalls)`** — сохраняет SYSTEM-сообщение с JSON-представлением tool calls (кроме скрытых инструментов: `recordChatInsights`, `getUserName`, `getCurrentDateTime`, `getOriginalMessages`). Использует `findFirstByConversationIdOrderByPositionDesc` для определения следующей позиции.

### 5. ChatTopicEntity — топик чата

`ChatTopicEntity` — сущность, представляющая одну беседу. Таблица `chat_topic`:

| Поле | Тип | Описание |
|---|---|---|
| `conversation_id` | `String` (PK) | UUID беседы |
| `user` | `String` | Владелец |
| `is_user` | `boolean` | true = пользователь, false = система |
| `topic` | `String` | Название/тема |
| `created_at` | `LocalDateTime` | Создан |
| `updated_at` | `LocalDateTime` | Обновлён |

**Авто-именование:** `TopicFunction` — AI-инструмент, который автоматически генерирует название чата на основе первого сообщения пользователя.

### 6. Стриминг (SSE)

Стриминг реализован через `SseEmitter` (Spring MVC) и Reactor `Flux`:

```
POST /api/chat/stream
    │
    ├── SseEmitter (таймаут 30 минут)
    │
    └── chatClient.prompt()
         .user(userMessage)
         .toolContext(...)
         .advisors(...)
         .stream()
         .chatResponse()
         .subscribe(
             onNext  → StreamMessage (текст + finishReason),
             onError → emitter.completeWithError(),
             onDone  → ToolCallsMessage + StreamMessage("", "DONE") + summarize
         )
```

**Типы SSE-сообщений (DTO):**

| DTO | Когда | Поля |
|---|---|---|
| `StreamMessage` | Каждый чанк текста | `message` (String), `finishReason` (String или null) |
| `ToolCallMessage` | Начало/конец вызова инструмента | `toolCall` (ToolInvocation) |
| `ToolCallsMessage` | Завершение ответа | `toolCalls` (список ToolInvocation) |

**ToolInvocation** — отдельный record:

| Поле | Тип | Описание |
|---|---|---|
| `name` | String | Имя инструмента |
| `arguments` | Map | Параметры вызова |
| `status` | String | `STARTED`, `OK`, `ERROR` |
| `error` | String | Сообщение об ошибке (при ERROR) |
| `resultGist` | String | Краткая сводка результата (из `CompactToolResultConverter`) |

### 7. RecordingToolCallback и ToolInvocationCollector

**RecordingToolCallback** — обёртка над каждым `ToolCallback`, которая:
1. Перехватывает вызов инструмента
2. Извлекает `ToolInvocationCollector` из `ToolContext`
3. Записывает `STARTED` → выполняет делегат → записывает `OK` или `ERROR`
4. Фильтрует аргументы (только из белого списка `ALLOWED_INPUT_ARGUMENTS`)

**ToolInvocationCollector** — потокобезопасный коллектор (`CopyOnWriteArrayList`):
- `record(invocation)` — добавляет запись и отправляет в liveSink (SSE)
- `snapshot()` — все вызовы
- `completedSnapshot()` — только завершённые (без STARTED)

### 8. Суммаризация (SummarizeService)

`SummarizeService` автоматически сжимает длинные диалоги, чтобы не превысить контекстное окно модели.

**Принцип работы:**
1. После каждого ответа AI вызывается `trySummarize(conversationId)` — асинхронно, на virtual thread
2. Блокировка через `Striped<Lock>` (1028 полос) — одна беседа не может суммаризироваться параллельно
3. Проверяются два порога:
   - **Количество сообщений:** ≥ 20 несуммаризированных
   - **Объём токенов:** ≥ 3000 оценочных токенов (1 токен ≈ 4 символа)
4. Если порог превышен — сообщения до `liveMessages.size() - OVERLAP_MESSAGES` (10) сжимаются
5. Существующие суммаризации загружаются как контекст
6. Если накоплено ≥ 5 суммаризаций — они схлопываются в одну мета-суммаризацию
7. LLM генерирует суммаризацию через отдельный `ChatClient` с промптом, загружаемым через `@Value("classpath:prompt/summarizer.md") Resource summarizerPrompt`
8. Старые сообщения помечаются `summarized = true`, новая суммаризация сохраняется как `SYSTEM` сообщение с `summary = true`

**Промпт суммаризатора** (`prompt/summarizer.md`):
- Роль: создавать плотные информативные резюме
- Сохранять: решения, факты, нерешённые вопросы, цели
- Цитировать сообщения как `[msg:XYZ]`
- Имеет доступ к `MessageLookupFunction` для получения полного текста сообщений

### 9. Jira-чаты (JiraChatService)

`JiraChatService` создаёт чаты, предзагруженные контекстом задачи Jira.

**Поток создания (`POST /api/chat/jira`):**
1. Парсинг JIRA URL → issue key
2. Получение содержимого задачи через `JiraService`
3. Создание `ChatTopicEntity` с авто-названием (issue key + summary)
4. Сохранение JIRA-контента как вложения (`.md`, `source_url`)
5. Опционально — получение и сохранение Confluence-страницы
6. Суммаризация всех вложений чата
7. Сохранение приветственного ASSISTANT-сообщения с контекстом задачи

**Обновление (`POST /api/chat/jira/{id}/refresh`):**
- Удаляет старые вложения
- Заново получает данные из Jira
- Пересоздаёт вложения и суммаризацию
- Сообщения чата не трогает

**DTO:**
- `CreateJiraChatRequest` — `jiraUrl` (обязательно), `confluenceUrl` (опционально), `title` (опционально)
- `Chat` — `conversationId`, `user`, `topic`, `createdAt`, `updatedAt`, `messages`

### 10. Промпты

| Файл | Назначение | Используется |
|---|---|---|
| `prompt/sys.md` | Системный промпт ассистента | `ChatClient` — через `@Value` + `Resource` (все запросы) |
| `prompt/summarizer.md` | Промпт суммаризатора | `SummarizeService` — через `@Value` + `Resource` |

**sys.md** определяет:
- Роль: «Ассистент для работы с базой знаний»
- Запрет на выдумку — всегда вызывать инструменты
- Скрытые инструменты (вызываются фоново, не упоминаются)
- Git-ограничения (read-only, tracked, усечение diff'ов, фрагментация больших файлов)
- Алгоритм ответа (вопрос → нужны ли данные → вызов инструмента → ответ)
- Таблицу выбора инструментов
- Правила цепочек вызовов
- Шпаргалку grepContent с примерами запросов

### 11. Поток данных при запросе

```
1. Frontend отправляет POST /api/chat/stream
   conversationId=UUID, body="текст вопроса"

2. ChatController.checkChat()
   └─ Создаёт ChatTopicEntity если новый conversationId
   └─ Обновляет updated_at

3. ChatClient.prompt()
   ├─ System prompt (prompt/sys.md, загружен через @Value Resource)
   ├─ ChatMemory (последние 50 сообщений)
   ├─ User message
   └─ ToolContext (conversationId, collector, userName)

4. Модель отвечает (потоково)
   ├─ Текст → StreamMessage → SSE
   ├─ Вызов инструмента → RecordingToolCallback → ToolInvocationCollector
   │   └─ ToolCallMessage (с resultGist) → SSE
   └─ Завершение → ToolCallsMessage + StreamMessage("", "DONE") → SSE

5. ChatMemoryService.saveToolCalls()
   └─ Сохраняет SYSTEM-сообщение с JSON tool calls (кроме скрытых)

6. SummarizeService.trySummarize() (асинхронно)
   └─ Проверяет пороги → при необходимости сжимает

7. ChatMemoryService сохраняет сообщения в chat_message

8. При обрыве SSE (disconnect):
   └─ .doFinally → persistPartial()
       └─ Сохраняет накопленный buffer как AssistantMessage + tool calls
```
