## Обзор чат-системы

Документ описывает архитектуру и ключевые компоненты чат-системы Knowledge Base. Чат — это AI-ассистент, который отвечает на вопросы пользователя, используя инструменты (tools) для доступа к базе знаний и Git-репозиторию.

**Связанные документы:**
- [API Reference](api-reference.md) — эндпоинты `ChatController`, `PhraseController`
- [AI-инструменты](ai-инструменты.md) — полный список инструментов
- [Модели данных — Чат](модели-данных/чат.md) — сущности `ChatTopicEntity`, `ChatMessageEntity`
- [Конфигурация](конфигурация.md) — `ChatConfig`, параметры AI, `ChatModelProperties`

---

### 1. Архитектура чата

```
Пользователь (Frontend)
    │  SSE / HTTP
    ▼
ChatController  (/api/chats)
    │
    ├── ChatClient (Spring AI)
    │       ├── ChatModel (OpenAI-совместимый)
    │       ├── ChatMemory → ChatMemoryService → PostgreSQL (chat_message)
    │       ├── ToolCallbacks (5 функций) → RecordingToolCallback → ToolInvocationCollector
    │       └── System Prompt (@Value classpath:prompt/sys.md)
    │
    ├── ChatTopicRepository → PostgreSQL (chat_topic)
    ├── ChatModelProperties — конфигурация доступных моделей
    ├── SummarizeService (фоновая суммаризация)
    └── JiraChatService (чат по задаче Jira)
```

**Ключевые особенности:**
- **Стриминг через SSE** — ответы передаются в реальном времени через `SseEmitter`
- **Инструменты (tools)** — AI может вызывать функции для поиска документов, чтения файлов, истории коммитов
- **Компактные результаты** — `CompactToolResultConverter` сжимает результаты в `resultGist` для отображения в UI
- **Сохранение tool calls** — `ChatMemoryService.saveToolCalls()` сохраняет SYSTEM-сообщение с JSON tool calls (кроме скрытых инструментов)
- **Частичное сохранение при обрыве SSE** — при разрыве соединения накопленный ответ сохраняется как `AssistantMessage`
- **Суммаризация** — длинные диалоги автоматически сжимаются, чтобы не превысить контекстное окно
- **Jira-чаты** — можно создать чат, предзагруженный контентом задачи Jira и страницы Confluence
- **Выбор модели** — пользователь может переключать AI-модель для каждого чата
- **Библиотека фраз** — готовые подсказки над полем ввода в пустом чате, управляются из БД через [PhraseController](api-reference.md)

### 2. ChatController — точка входа

`ChatController` (`/api/chats`) — контроллер чата. Зависимости:

| Зависимость | Назначение |
|---|---|
| `ChatClient` | Основной клиент Spring AI для общения с моделью |
| `ChatMemory` | Управление историей сообщений (скользящее окно) |
| `ChatTopicRepository` | CRUD для топиков чата |
| `ChatMemoryService` | Персистентное хранение сообщений в БД |
| `ChatModelProperties` | Конфигурация доступных моделей (из `kb.chat.*`) |
| `SummarizeService` | Фоновая суммаризация длинных диалогов |
| `JiraChatService` | Создание чатов с контекстом Jira |

**Эндпоинты** (подробнее в [API Reference](api-reference.md)):
- `POST /api/chats/{conversationId}/messages/stream` — стриминг ответа AI
- `POST /api/chats/{conversationId}/messages/test` — тестовый стриминг (демонстрация tool calls)
- `POST /api/chats/{conversationId}/messages` — синхронный запрос
- `GET /api/chats` — список чатов пользователя
- `GET /api/chats/{conversationId}` — конкретный чат с сообщениями (параметр `includeMessages`)
- `DELETE /api/chats/{conversationId}` — удаление чата
- `PUT /api/chats/{conversationId}/topic` — установка/обновление топика
- `POST /api/chats/jira` — создание Jira-чата
- `POST /api/chats/{conversationId}/refresh` — обновление Jira-чата
- `GET /api/chats/models` — список доступных моделей и дефолтная
- `PUT /api/chats/{conversationId}/model` — задать модель чата (пустое тело → сброс к дефолтной)

При загрузке истории (`GET /api/chats/{conversationId}`) фильтруются сообщения типа `SYSTEM` — они не показываются пользователю.

#### Частичное сохранение при обрыве SSE

При разрыве соединения (клиент закрыл вкладку, сетевой сбой) `ChatController` сохраняет накопленный ответ ассистента:

- **`StringBuffer buffer`** — накапливает чанки текста ответа ассистента
- **`AtomicBoolean persisted`** — защита от двойного сохранения
- **`.doFinally(signal)`** — при `CANCEL` или `ON_ERROR` вызывает `persistPartial()`
- **`persistPartial()`** — сохраняет накопленный текст как `AssistantMessage` + tool calls через `chatMemoryService.saveToolCalls()`
- **Сброс буфера:** при `finishReason != null` (конец сегмента) буфер очищается (`buffer.setLength(0)`) — каждый сегмент сохраняется отдельно
- **`emitter.onCompletion()`** вызывает `dispose()` — разрыв клиентом → `CANCEL` → частичное сохранение

### 3. Выбор AI-модели

Пользователь может выбирать модель для каждого чата. Конфигурация доступных моделей задаётся через `ChatModelProperties` (префикс `kb.chat` в `application.yaml`).

#### ChatModelProperties

```java
@ConfigurationProperties(prefix = "kb.chat")
public record ChatModelProperties(
    ModelOption defaultModel,
    List<ModelOption> models
) {
    public record ModelOption(String id, String label) {}
    public boolean isAllowed(String id) { ... }
}
```

**Пример конфигурации в `application.yaml`:**
```yaml
kb:
  chat:
    default-model:
      id: ${AI_MODEL:gpt-4o}
      label: ${AI_MODEL_LABEL:GPT-4o}
    models:
      - id: ${KB_CHAT_MODELS_0_ID:gpt-4o-mini}
        label: ${KB_CHAT_MODELS_0_LABEL:GPT-4o Mini}
      - id: ${KB_CHAT_MODELS_1_ID:claude-3-opus}
        label: ${KB_CHAT_MODELS_1_LABEL:Claude 3 Opus}
```

#### Приоритет выбора модели (`resolveModel()`)

При каждом запросе модель определяется по цепочке приоритетов:

1. **Параметр запроса `?model=`** — явно указанная модель в `streamMessage()` / `createMessage()`. Если указана — валидируется через `isAllowed()`, сохраняется как «последняя» для чата, и используется.
2. **Сохранённая модель чата** — `ChatTopicEntity.model` (последняя использованная модель для этого чата). Проверяется через `isAllowed()` на случай, если модель убрали из конфига.
3. **Дефолтная модель** — `ChatModelProperties.defaultModel.id` (из `application.yaml`).

#### API для моделей

Модели находятся в `ChatController`:

| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/chats/models` | Возвращает `ChatModelProperties` (список доступных моделей) |
| `PUT` | `/api/chats/{conversationId}/model` | Задать модель чата. Пустое тело → сброс к дефолтной. `400` если модель не в списке разрешённых |

#### Связь со Spring AI

`spring.ai.openai.chat.options.model` ссылается на `${kb.chat.default-model.id}` — это модель по умолчанию для `ChatModel`. При явном указании модели в запросе она передаётся через `ChatOptions` в `ChatClient.prompt().options(...)`.

### 4. ChatClient и ChatConfig

`ChatConfig` — центральная конфигурация чата. Создаёт два бина:

#### 4.1 ChatMemory
```java
MessageWindowChatMemory.builder()
    .chatMemoryRepository(chatMemoryService)  // персистентное хранение
    .maxMessages(50)                          // скользящее окно
    .build();
```
- Хранит последние 50 сообщений в оперативной памяти
- При превышении — старые сообщения вытесняются (но сохраняются в БД через `ChatMemoryService`)

#### 4.2 ChatClient
- **ChatModel** — OpenAI-совместимая модель (через `OpenAiChatModel`)
- **System Prompt** — загружается через `@Value("classpath:prompt/sys.md") Resource sysPrompt` и передаётся в `.defaultSystem(sysPrompt)`
- **MessageChatMemoryAdvisor** — автоматически добавляет историю в каждый запрос
- **ToolCallingManager** — `ToolCallingManager.builder().toolCallbacks(...)` с 5 ToolCallbacks, обёрнутых в `RecordingToolCallback`:
  - `TopicFunction` — авто-именование чата
  - `MessageLookupFunction` — поиск сообщений по позиции
  - `DocumentFunction` — работа с документами базы знаний
  - `GitFunction` — работа с Git-репозиторием
  - `AttachmentFunction` — работа с вложениями

### 5. ChatMemoryService — персистентность сообщений

`ChatMemoryService` реализует `ChatMemoryRepository` и отвечает за сохранение сообщений в PostgreSQL.

**Таблица `chat_message`** (подробнее в [Модели данных — Чат](модели-данных/чат.md)):
| Поле | Тип | Описание |
|---|---|---|
| `id` | `long` | Первичный ключ |
| `conversation_id` | `String` | ID беседы |
| `content` | `String` | Текст сообщения |
| `type` | `MessageType` | USER, ASSISTANT, SYSTEM, TOOL |
| `position` | `long` | Порядковый номер в беседе |
| `summarized` | `boolean` | Суммаризировано ли |
| `summary` | `boolean` | Является ли само сообщение суммаризацией |
| `meta` | `String` (JSON) | Метаданные tool calls (`ChatMessageMeta` → JSON через `ChatMessageMetaToJsonConverter`) |
| `created_at` | `LocalDateTime` | Время создания |

**Ключевые методы:**
- `saveAll` — сохраняет новые сообщения, фильтруя уже существующие (по `IMessage`)
- `findByConversationId` — возвращает несуммаризированные сообщения
- `findChatMessageByConversationId` — возвращает не-summary сообщения (для отображения пользователю), фильтрует `MessageType.SYSTEM`
- **`saveToolCalls(conversationId, toolCalls)`** — сохраняет SYSTEM-сообщение с JSON-представлением tool calls (кроме скрытых инструментов: `recordChatInsights`, `getUserName`, `getCurrentDateTime`, `getOriginalMessages`). Использует `findFirstByConversationIdOrderByPositionDesc` для определения следующей позиции.

### 7. Стриминг (SSE)

Стриминг реализован через `SseEmitter` (Spring MVC) и Reactor `Flux`:

```
POST /api/chats/{conversationId}/messages/stream
    │
    ├── SseEmitter (таймаут 30 минут)
    │
    └── chatClient.prompt()
         .user(userMessage)
         .toolContext(...)
         .advisors(...)
         .stream()
         .chatResponse()
         .subscribe(
             onNext  → StreamMessage (текст + finishReason),
             onError → emitter.completeWithError(),
             onDone  → ToolCallsMessage + StreamMessage("", "DONE") + summarize
         )
```

**Типы SSE-сообщений (DTO):**

| DTO | Когда | Поля |
|---|---|---|
| `StreamMessage` | Каждый чанк текста | `message` (String), `finishReason` (String или null) |
| `ToolCallMessage` | Начало/конец вызова инструмента | `toolCall` (ToolInvocation) |
| `ToolCallsMessage` | Завершение ответа | `toolCalls` (список ToolInvocation) |

**ToolInvocation** — запись о вызове инструмента (runtime, с `resultGist`):

| Поле | Тип | Описание |
|---|---|---|
| `name` | String | Имя инструмента |
| `arguments` | Map | Параметры вызова |
| `status` | String | `STARTED`, `OK`, `ERROR` |
| `error` | String | Сообщение об ошибке (при ERROR) |
| `resultGist` | String | Краткая сводка результата (из `CompactToolResultConverter`) |
| `resultMeta` | Object | Метаданные результата (через `ToolCallResultMetaProvider`) |

**ToolInvocationMeta** — облегчённая версия для хранения в БД и отправки на фронтенд (без `resultGist`). Содержит `name`, `arguments`, `status`, `error`, `resultMeta`.

### 8. RecordingToolCallback и ToolInvocationCollector

**RecordingToolCallback** — обёртка над каждым `ToolCallback`, которая:
1. Перехватывает вызов инструмента
2. Извлекает `ToolInvocationCollector` из `ToolContext`
3. Записывает `STARTED` → выполняет делегат → записывает `OK` или `ERROR`
4. Извлекает `resultMeta` через `ToolCallResultMetaProvider` (если результат его реализует)
5. Вычисляет `resultGist` самостоятельно (из `resultMeta` или через `CompactToolResultConverter`)
6. Фильтрует аргументы (только из белого списка `ALLOWED_INPUT_ARGUMENTS`)

**ToolInvocationCollector** — потокобезопасный коллектор (`CopyOnWriteArrayList`):
- `record(invocation)` — добавляет запись и отправляет в liveSink (SSE)
- `snapshot()` — все вызовы
- `completedSnapshot()` — только завершённые (без STARTED)

### 9. Суммаризация (SummarizeService)

`SummarizeService` автоматически сжимает длинные диалоги, чтобы не превысить контекстное окно модели.

**Принцип работы:**
1. После каждого ответа AI вызывается `trySummarize(conversationId)` — асинхронно, на virtual thread
2. Блокировка через `Striped<Lock>` (1028 полос) — одна беседа не может суммаризироваться параллельно
3. Проверяются два порога:
   - **Количество сообщений:** ≥ 20 несуммаризированных
   - **Объём токенов:** ≥ 3000 оценочных токенов (1 токен ≈ 4 символа)
4. Если порог превышен — сообщения до `liveMessages.size() - OVERLAP_MESSAGES` (10) сжимаются
5. Существующие суммаризации загружаются как контекст
6. Если накоплено ≥ 5 суммаризаций — они схлопываются в одну мета-суммаризацию
7. LLM генерирует суммаризацию через отдельный `ChatClient` с промптом, загружаемым через `@Value("classpath:prompt/summarizer.md") Resource summarizerPrompt`
8. Старые сообщения помечаются `summarized = true`, новая суммаризация сохраняется как `SYSTEM` сообщение с `summary = true`

**Промпт суммаризатора** (`prompt/summarizer.md`):
- Роль: создавать плотные информативные резюме
- Сохранять: решения, факты, нерешённые вопросы, цели
- Цитировать сообщения как `[msg:XYZ]`
- Имеет доступ к `MessageLookupFunction` для получения полного текста сообщений

### 10. Jira-чаты (JiraChatService)

`JiraChatService` создаёт чаты, предзагруженные контекстом задачи Jira.

**Поток создания (`POST /api/chats/jira`):**
1. Парсинг JIRA URL → issue key
2. Получение содержимого задачи через `JiraService`
3. Создание `ChatTopicEntity` с авто-названием (issue key + summary)
4. Сохранение JIRA-контента как вложения (`.md`, `source_url`)
5. Опционально — получение и сохранение Confluence-страницы
6. Суммаризация всех вложений чата
7. Сохранение приветственного ASSISTANT-сообщения с контекстом задачи

**Обновление (`POST /api/chats/{conversationId}/refresh`):**
- Удаляет старые вложения
- Заново получает данные из Jira
- Пересоздаёт вложения и суммаризацию
- Сообщения чата не трогает

**DTO:**
- `CreateJiraChatRequest` — `jiraUrl` (обязательно), `confluenceUrl` (опционально), `title` (опционально)
- `Chat` — `conversationId`, `user`, `topic`, `createdAt`, `updatedAt`, `messages`

### 11. Библиотека фраз (PhraseController)

Фразы — готовые подсказки над полем ввода в пустом чате. Раньше были захардкожены в `GIT_PHRASES`, теперь хранятся в БД (таблица `phrase`) и управляются через админку.

**Публичные эндпоинты** (`/api/phrases`):
- `GET /api/phrases` — только enabled, `ORDER BY category, position`
- `PATCH /api/phrases/{id}/favorite?value=` — переключить избранное

**Админские эндпоинты** (`/api/admin/phrases`):
- `GET /api/admin/phrases?q=` — все фразы (вкл. выключенные), `q` — быстрый поиск по label (ILIKE)
- `POST /api/admin/phrases` — создать (добавляется в конец категории)
- `PUT /api/admin/phrases/{id}` — изменить (смена категории → переезд в конец новой)
- `DELETE /api/admin/phrases/{id}` — удалить
- `PATCH /api/admin/phrases/{id}/favorite?value=` — тоггл избранного
- `PATCH /api/admin/phrases/{id}/enabled?value=` — тоггл enabled
- `PATCH /api/admin/phrases/{id}/move` — переупорядочить; тело `{ "position": <слот соседа> }`

**Сущность `PhraseEntity`** (таблица `phrase`):
| Поле | Тип | Описание |
|---|---|---|
| `id` | Long | PK |
| `category` | String | Произвольная категория |
| `label` | String | Название (на чипе) |
| `text` | String | Текст, вставляемый в поле ввода |
| `position` | int | Порядок внутри категории |
| `enabled` | boolean | Показывать в чате |
| `favorite` | boolean | Избранное |
| `createdAt` | Instant | |
| `updatedAt` | Instant | |

**Move** переиспользует windowed-shift из `DocumentRepository` (`shiftWindowUp`/`shiftWindowDown`), но фильтрует по `category = :category`. Зазоры в позициях допускаются.

**Фронтенд:**
- `Phrases.jsx` — блок фраз в чате (замена `GitPhrases.jsx`)
- `PhrasesSettings.jsx` — админская вкладка в SettingsPanel (полный CRUD + стрелки ↑/↓ + поиск)
- `phrasesApi.js` — обёртка над REST-эндпоинтами

### 12. Промпты

| Файл | Назначение | Используется |
|---|---|---|
| `prompt/sys.md` | Системный промпт ассистента | `ChatClient` — через `@Value` + `Resource` (все запросы) |

- Алгоритм ответа (вопрос → нужны ли данные → вызов инструмента → ответ)
- Таблицу выбора инструментов
- Правила цепочек вызовов
- Шпаргалку grepContent с примерами запросов

### 13. Поток данных при запросе

```
1. Frontend отправляет POST /api/chats/{conversationId}/messages/stream
   conversationId=UUID, body="текст вопроса", ?model= (опционально)

2. ChatController.checkChat()
   └─ Создаёт ChatTopicEntity если новый conversationId
   └─ Обновляет updated_at

3. resolveModel() — определяет модель по цепочке:
   параметр запроса → сохранённая модель чата → дефолтная

4. ChatClient.prompt()
   ├─ System prompt (prompt/sys.md, загружен через @Value Resource)
   ├─ ChatMemory (последние 50 сообщений)
   ├─ User message
   └─ ToolContext (conversationId, collector, userName)

5. Модель отвечает (потоково)
   ├─ Текст → StreamMessage → SSE
   ├─ Вызов инструмента → RecordingToolCallback → ToolInvocationCollector
   │   └─ ToolCallMessage (с resultGist + resultMeta) → SSE
   └─ Завершение → ToolCallsMessage + StreamMessage("", "DONE") → SSE

6. ChatMemoryService.saveToolCalls()
   └─ Сохраняет SYSTEM-сообщение с JSON tool calls (кроме скрытых)

7. SummarizeService.trySummarize() (асинхронно)
   └─ Проверяет пороги → при необходимости сжимает

8. ChatMemoryService сохраняет сообщения в chat_message

9. При обрыве SSE (disconnect):
   └─ .doFinally → persistPartial()
       └─ Сохраняет накопленный buffer как AssistantMessage + tool calls
```

|---|---|
| `controller/ChatController.java` | Эндпоинты чата, стриминг, модели, частичное сохранение при обрыве SSE |
| `controller/PhraseController.java` | Эндпоинты библиотеки фраз (публичные + админские) |
| `config/ChatConfig.java` | Конфигурация ChatClient, ChatMemory, ToolCallingManager |
| `config/model/ChatModelProperties.java` | Конфигурация доступных моделей (`kb.chat.*`) |
| `service/ChatMemoryService.java` | Персистентность сообщений, saveToolCalls() |
| `service/SummarizeService.java` | Суммаризация диалогов |
| `service/JiraChatService.java` | Jira-чаты |
| `service/PhraseService.java` | Бизнес-логика библиотеки фраз |
| `model/chat/entity/ChatTopicEntity.java` | Сущность топика (включая поле `model`) |
| `model/chat/entity/ChatMessageEntity.java` | Сущность сообщения (включая поле `meta`) |
| `model/phrase/entity/PhraseEntity.java` | Сущность фразы |
| `model/phrase/dto/Phrase.java` | DTO фразы |
| `model/phrase/dto/PhraseRequest.java` | DTO создания/обновления фразы |
| `model/phrase/dto/MovePhraseRequest.java` | DTO перемещения фразы |
| `model/chat/dto/Chat.java` | DTO чата |
| `model/chat/dto/ChatMessage.java` | DTO сообщения |
| `model/chat/dto/StreamMessage.java` | DTO стрим-чанка |
| `model/chat/dto/ToolCallMessage.java` | DTO вызова инструмента |
| `model/chat/dto/ToolCallsMessage.java` | DTO списка вызовов |
| `model/chat/dto/CreateJiraChatRequest.java` | DTO запроса Jira-чата |
| `model/chat/dto/ToolInvocation.java` | Record вызова инструмента (name, arguments, status, error, resultGist, resultMeta) |
| `model/chat/dto/ToolInvocationMeta.java` | Облегчённая версия для БД и фронтенда (без resultGist) |
| `model/chat/dto/ChatMessageMeta.java` | Обёртка над `List<ToolInvocationMeta>` |
| `model/chat/convert/ChatMessageMetaToJsonConverter.java` | JPA-конвертер `ChatMessageMeta` ↔ JSON |
| `tools/RecordingToolCallback.java` | Обёртка для записи вызовов |
| `tools/ToolInvocationCollector.java` | Коллектор вызовов |
| `tools/CompactToolResultConverter.java` | Конвертер результатов в компактное превью |
| `tools/ToolCallResultMetaProvider.java` | Интерфейс для DTO, возвращающих `resultMeta` |
| `functions/MessageLookupFunction.java` | Инструмент поиска сообщений |
| `repository/ChatTopicRepository.java` | Репозиторий топиков |
| `repository/ChatMessageRepository.java` | Репозиторий сообщений |
| `repository/PhraseRepository.java` | Репозиторий фраз |
| `utils/ChatUtils.java` | Утилиты (buildContext, getUser) |
| `resources/prompt/sys.md` | Системный промпт |
| `resources/prompt/summarizer.md` | Промпт суммаризатора |
| `resources/db/migration/V2026.06.12_00__create_phrase.sql` | Миграция: таблица `phrase` + сид |
| `frontend/src/components/chatPanel/Phrases.jsx` | Блок фраз в чате |
| `frontend/src/components/chatPanel/phrasesApi.js` | API-обёртка для фраз |
| `frontend/src/components/settingsPanel/PhrasesSettings.jsx` | Админка фраз в SettingsPanel |
