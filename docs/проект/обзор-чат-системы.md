## Обзор чат-системы

Документ описывает архитектуру и ключевые компоненты чат-системы Knowledge Base. Чат — это AI-ассистент, который отвечает на вопросы пользователя, используя инструменты (tools) для доступа к базе знаний и Git-репозиторию.

**Связанные документы:**
- [API Reference](api-reference.md) — эндпоинты `ChatController`, `GitController`, `PhraseController`, `SettingsController`
- [AI-инструменты](ai-инструменты.md) — полный список инструментов
- [Модели данных — Чат](модели-данных/чат.md) — сущности `ChatTopicEntity`, `ChatMessageEntity`
- [Конфигурация](конфигурация.md) — `ChatConfig`, параметры AI, `ChatModelProperties`, `SecurityConfig`, `SubAgentConfig`

---

### 1. Архитектура чата

```
Пользователь (Frontend)
    │  HTTP (REST + SSE events)
    ▼
ChatController  (/api/chats)
    │
    ├── ChatRunService — запуск фоновой генерации (virtual threads)
    │       ├── ChatClient (Spring AI)
    │       │     ├── ChatModel (OpenAI-совместимый)
    │       │     ├── ChatMemory → ChatMemoryService → PostgreSQL (chat_message)
    │       │     ├── ToolCallbacks (6 функций) → RecordingToolCallback → ToolInvocationCollector
    │       │     └── System Prompt (@Value classpath:prompt/sys.md)
    │       └── ChatEventService → ConversationHub (in-memory pub/sub + replay)
    │
    ├── ChatEventService — SSE-канал: стриминг + кросс-вкладочная синхронизация
    │       └── ConversationHub — лог событий + fan-out на подписчиков (SseEmitter)
    │
    ├── ChatRuntimeMonitor — мониторинг утечек (логирование размеров реестров)
    ├── ChatTopicRepository → PostgreSQL (chat_topic)
    ├── ChatModelProperties — конфигурация доступных моделей
    ├── SummarizeService (фоновая суммаризация)
    └── JiraChatService (чат по задаче Jira)

GitController  (/api/git)
    └── GitService — fuzzy-поиск и чтение tracked-файлов репозитория

SettingsController  (/api/settings)
    └── GET /api/settings/ai-config — снапшот AI-конфигурации для панели настроек

SecurityConfig — HTTP Basic Auth (stateless, in-memory user)
```

**Ключевые особенности:**
- **Фоновые прогоны (background runs)** — генерация ответа не привязана к HTTP-запросу: `POST /runs` запускает задачу и сразу возвращает `runId`, ответ приходит через SSE event channel. Ответ переживает перезагрузку страницы и виден всем вкладкам
- **SSE event channel** — единый поток событий (`GET /events`) для стриминга ответа и кросс-вкладочной синхронизации. При подключении реплеятся пропущенные события (по `seq`), затем идут живые
- **Остановка генерации** — явный сигнал `POST /runs/{runId}/stop` вместо разрыва соединения
- **Инструменты (tools)** — AI может вызывать функции для поиска документов, чтения файлов, истории коммитов, а также `searchCodebase` (sub-agent для многошагового поиска)
- **Компактные результаты** — `CompactToolResultConverter` сжимает результаты в `resultGist` для отображения в UI
- **Сохранение tool calls** — `ChatMemoryService.saveToolCalls()` сохраняет SYSTEM-сообщение с JSON tool calls (кроме скрытых инструментов)
- **Частичное сохранение при остановке/ошибке** — при `RUN_STOPPED`/`RUN_ERROR` накопленный ответ сохраняется с маркером `[stopped]`/`[error]`
- **Суммаризация** — длинные диалоги автоматически сжимаются, чтобы не превысить контекстное окно
- **Jira-чаты** — можно создать чат, предзагруженный контентом задачи Jira и страницы Confluence
- **Выбор модели** — пользователь может переключать AI-модель для каждого чата
- **Библиотека фраз** — готовые подсказки над полем ввода в пустом чате, управляются из БД через [PhraseController](api-reference.md)
- **Вставка файлов репозитория (`/file`)** — fuzzy-поиск tracked-файлов прямо в композере чата, вставка как collapsible chips с превью содержимого
- **HTTP Basic Auth** — stateless-аутентификация с единственным in-memory пользователем из конфига (`kb.security.*`)
- **searchCodebase sub-agent** — инструмент для многошагового поиска: изолированный mini-agent с собственным system prompt и ограниченным набором read-only инструментов

### 2. ChatController — точка входа

`ChatController` (`/api/chats`) — контроллер чата. Зависимости:

| Зависимость | Назначение |
|---|---|
| `ChatClient` | Основной клиент Spring AI для общения с моделью |
| `ChatMemory` | Управление историей сообщений (скользящее окно) |
| `ChatTopicRepository` | CRUD для топиков чата |
| `ChatMemoryService` | Персистентное хранение сообщений в БД |
| `ChatModelProperties` | Конфигурация доступных моделей (из `kb.chat.*`) |
| `JiraChatService` | Создание чатов с контекстом Jira |
| `ChatRunService` | Запуск фоновой генерации ответа |
| `ChatEventService` | Публикация событий в SSE-канал |

**Эндпоинты** (подробнее в [API Reference](api-reference.md)):
- `POST /api/chats/{conversationId}/runs` — запустить фоновую генерацию (возвращает `runId`)
- `GET /api/chats/{conversationId}/events` — SSE-поток событий чата (стриминг + синхронизация)
- `POST /api/chats/{conversationId}/runs/{runId}/stop` — остановить активный прогон
- `GET /api/chats/{conversationId}/runs/active` — runId активного прогона (или `{}`)
- `POST /api/chats/{conversationId}/messages` — синхронный запрос
- `GET /api/chats` — список чатов пользователя
- `GET /api/chats/{conversationId}` — конкретный чат с сообщениями (параметр `includeMessages`)
- `DELETE /api/chats/{conversationId}` — удаление чата
- `PUT /api/chats/{conversationId}/topic` — установка/обновление топика
- `POST /api/chats/jira` — создание Jira-чата
- `POST /api/chats/{conversationId}/refresh` — обновление Jira-чата
- `GET /api/chats/models` — список доступных моделей и дефолтная
- `PUT /api/chats/{conversationId}/model` — задать модель чата (пустое тело → сброс к дефолтной)

При загрузке истории (`GET /api/chats/{conversationId}`) фильтруются сообщения типа `SYSTEM` — они не показываются пользователю.

#### Фоновые прогоны (Background Runs)

Генерация ответа больше не привязана к HTTP-запросу:

1. **`POST /runs`** — принимает сообщение, запускает генерацию в фоновом потоке (virtual thread) и сразу возвращает `{ "runId": "..." }`. Если генерация уже идёт — `409 Conflict`
2. **`GET /events`** — постоянное SSE-соединение. При подключении реплеит пропущенные события (по `fromSeq`), затем шлёт живые. События: `USER_MESSAGE`, `RUN_STARTED`, `STREAM`, `TOOL_CALL`, `TOOL_CALLS`, `RUN_DONE`, `RUN_STOPPED`, `RUN_ERROR`
3. **`POST /runs/{runId}/stop`** — явная остановка: `dispose()` → `CANCEL` → частичное сохранение + событие `RUN_STOPPED`
4. **`GET /runs/active`** — для восстановления состояния после перезагрузки страницы

**Преимущества:**
- Ответ продолжает генерироваться после перезагрузки страницы
- Все вкладки видят один и тот же ответ (кросс-вкладочная синхронизация)
- Остановка — явный сигнал, а не разрыв соединения
- `clientMsgId` защищает от задвоения оптимистичного пузыря

### 3. ChatRunService — фоновые прогоны

`ChatRunService` — центральный сервис, запускающий генерацию как фоновую задачу:

- **`start(conversationId, user, userMessage, resolvedModel, clientMsgId)`** — атомарная заявка на чат (`activeByConversation`), запуск в `chatRunExecutor` (virtual threads + `DelegatingSecurityContextExecutorService`)
- **`stop(conversationId, runId)`** — `dispose()` реактивного стрима
- **`activeRun(conversationId)`** — делегирует в `ChatEventService`

**Поток выполнения (`run()`):**
1. Публикует `USER_MESSAGE` (эхо для других вкладок) и `RUN_STARTED`
2. Создаёт `ChatClient` prompt с `toolContext` (явная передача пользователя — на случай потоков Reactor)
3. Подписывается на реактивный стрим: `onNext` → `STREAM`/`TOOL_CALL`, `onComplete` → `TOOL_CALLS` + `RUN_DONE` + суммаризация
4. `doFinally`: `CANCEL` → частичное сохранение + `RUN_STOPPED`, `ON_ERROR` → частичное сохранение + `RUN_ERROR`
5. `cleanup()`: закрывает хаб прогона, снимает заявку

**Частичное сохранение:** при остановке/ошибке накопленный текст сохраняется с маркером `[stopped]`/`[error]` — после reload видно, что генерация была прервана.

### 4. ChatEventService и ConversationHub — SSE event channel

**ChatEventService** — in-memory pub/sub поверх `ConversationHub`:

- **`subscribe(conversationId, fromSeq)`** — создаёт/находит хаб, подписывает `SseEmitter` с replay пропущенных событий
- **`publish(conversationId, type, runId, clientMsgId, payload)`** — публикует событие всем подписчикам
- **`startRun`/`endRun`** — управление жизненным циклом прогона в хабе
- **`closeIfIdle()`** — автоматическая выгрузка простаивающего хаба (нет подписчиков и прогона)

**ConversationHub** — канал одного чата:

- **Лог событий** (`eventLog`) — хранит события только активного прогона, очищается при старте и завершении
- **Подписчики** (`subscribers`) — `SseEmitter` всех открытых вкладок
- **Replay** — при подключении с `fromSeq` реплеятся все события с `seq > fromSeq`
- **Потокобезопасность** — `ReentrantLock` (не `synchronized`, чтобы избежать pinning на виртуальных потоках)
- **Закрытие** — `closed` флаг предотвращает гонку «выгрузка ↔ новая подписка»

**ChatEvent** — запись одного события:
| Поле | Тип | Описание |
|---|---|---|
| `seq` | long | Монотонный номер (точка дозагрузки) |
| `type` | ChatEventType | Тип события |
| `runId` | String | К какому прогону относится |
| `clientMsgId` | String | ID клиента-инициатора (защита от задвоения) |
| `payload` | Object | Полезная нагрузка |

**ChatEventType:**
- `USER_MESSAGE` — эхо сообщения для других вкладок (payload: `UserMessagePayload`)
- `RUN_STARTED` — начало генерации
- `STREAM` — чанк текста (payload: `StreamMessage`)
- `TOOL_CALL` — обновление вызова инструмента (payload: `ToolCallMessage`)
- `TOOL_CALLS` — итоговый список вызовов сегмента (payload: `ToolCallsMessage`)
- `RUN_DONE` — успешное завершение
- `RUN_STOPPED` — остановлена пользователем
- `RUN_ERROR` — прервана ошибкой

### 7. SearchAgentService — sub-agent поиска

`SearchAgentService` — изолированный mini-agent, доступный основному чат-модели как инструмент `searchCodebase`:

- **Изоляция:** нет chat memory, только `task` строка на вход
- **Инструменты:** read-only подмножество git/document tools (из `kb.search.subagent.allowed-tools`)
- **Рекурсивная защита:** `searchCodebase` исключён из allowed-tools по построению
- **System prompt:** `prompt/search-agent.md` — специалист по поиску
- **Ручной цикл:** через `ToolCallingManager` с жёстким лимитом итераций (`maxIterations`)
- **При превышении лимита:** принудительная суммаризация (tool-less вызов)
- **Самокоррекция:** ошибки инструментов возвращаются модели для исправления
- **Результат:** `SearchAgentResult` — `report`, `complete`, `iterations`, `durationMs` (скрыто от модели)

**SubAgentConfig** (`kb.search.subagent.*`):
| Параметр | Описание |
|---|---|
| `enabled` | Включить/отключить sub-agent |
| `model-id` | Модель (может отличаться от основной) |
| `max-tokens` | Макс. токенов на вызов |
| `max-iterations` | Жёсткий лимит раундов |
| `allowed-tools` | Read-only инструменты (без `searchCodebase`) |

### 8. ChatMemoryService — персистентность сообщений

`ChatMemoryService` реализует `ChatMemoryRepository` и отвечает за сохранение сообщений в PostgreSQL.

**Таблица `chat_message`** (подробнее в [Модели данных — Чат](модели-данных/чат.md)):
| Поле | Тип | Описание |
|---|---|---|
| `id` | `long` | Первичный ключ |
| `conversation_id` | `String` | ID беседы |
| `content` | `String` | Текст сообщения |
| `type` | `MessageType` | USER, ASSISTANT, SYSTEM, TOOL |
| `position` | `long` | Порядковый номер в беседе |
| `summarized` | `boolean` | Суммаризировано ли |
| `summary` | `boolean` | Является ли само сообщение суммаризацией |
| `meta` | `String` (JSON) | Метаданные tool calls (`ChatMessageMeta` → JSON через `ChatMessageMetaToJsonConverter`) |
| `created_at` | `LocalDateTime` | Время создания |

**Ключевые методы:**
- `saveAll` — сохраняет новые сообщения, фильтруя уже существующие (по `IMessage`)
- `findByConversationId` — возвращает несуммаризированные сообщения
- `findChatMessageByConversationId` — возвращает не-summary сообщения (для отображения пользователю), фильтрует `MessageType.SYSTEM`
- **`saveToolCalls(conversationId, toolCalls)`** — сохраняет SYSTEM-сообщение с JSON-представлением tool calls (кроме скрытых инструментов)

### 10. Стриминг (SSE) — новый подход

Старый подход (`POST /messages/stream` с прямым SseEmitter) заменён на background runs + event channel:

```
POST /api/chats/{conversationId}/runs
    │
    ├── ChatRunService.start()
    │     ├── Атомарная заявка на чат (409 если уже идёт)
    │     ├── Публикация USER_MESSAGE + RUN_STARTED
    │     └── executor.execute(() -> run())
    │           ├── chatClient.prompt().stream().chatResponse()
    │           ├── onNext  → STREAM / TOOL_CALL → ChatEventService.publish()
    │           ├── onComplete → TOOL_CALLS + RUN_DONE + summarize
    │           └── doFinally → частичное сохранение + RUN_STOPPED/RUN_ERROR
    │
    └── Возвращает { "runId": "..." }

GET /api/chats/{conversationId}/events?fromSeq=0
    │
    └── ChatEventService.subscribe()
          ├── Replay пропущенных событий (seq > fromSeq)
          └── Живые события → SseEmitter.send()
```

**Типы SSE-сообщений (DTO):**

| DTO | Когда | Поля |
|---|---|---|
| `StreamMessage` | Каждый чанк текста | `message` (String), `finishReason` (String или null) |
| `ToolCallMessage` | Начало/конец вызова инструмента | `toolCall` (ToolInvocation) |
| `ToolCallsMessage` | Завершение сегмента | `toolCalls` (список ToolInvocationMeta) |
| `ChatEvent` | Любое событие в канале | `seq`, `type`, `runId`, `clientMsgId`, `payload` |

### 11. RecordingToolCallback и ToolInvocationCollector

**RecordingToolCallback** — обёртка над каждым `ToolCallback`, которая:
1. Перехватывает вызов инструмента
2. Извлекает `ToolInvocationCollector` из `ToolContext`
3. Записывает `STARTED` → выполняет делегат → записывает `OK` или `ERROR`
4. Извлекает `resultMeta` через `ToolCallResultMetaProvider` (если результат его реализует)
5. Вычисляет `resultGist` самостоятельно (из `resultMeta` или через `CompactToolResultConverter`)
6. Фильтрует аргументы (только из белого списка `ALLOWED_INPUT_ARGUMENTS`)

**ToolInvocationCollector** — потокобезопасный коллектор (`CopyOnWriteArrayList`):
- `record(invocation)` — добавляет запись и отправляет в liveSink (SSE)
- `snapshot()` — все вызовы
- `completedSnapshot()` — только завершённые (без STARTED)

### 12. Суммаризация (SummarizeService)

`SummarizeService` автоматически сжимает длинные диалоги, чтобы не превысить контекстное окно модели.

**Принцип работы:**
1. После каждого ответа AI вызывается `trySummarize(conversationId)` — асинхронно, на virtual thread
2. Блокировка через `Striped<Lock>` (1028 полос) — одна беседа не может суммаризироваться параллельно
3. Проверяются два порога:
   - **Количество сообщений:** ≥ 20 несуммаризированных
   - **Объём токенов:** ≥ 3000 оценочных токенов (1 токен ≈ 4 символа)
4. Если порог превышен — сообщения до `liveMessages.size() - OVERLAP_MESSAGES` (10) сжимаются
5. Существующие суммаризации загружаются как контекст
6. Если накоплено ≥ 5 суммаризаций — они схлопываются в одну мета-суммаризацию
7. LLM генерирует суммаризацию через отдельный `ChatClient` с промптом `prompt/summarizer.md`
8. Старые сообщения помечаются `summarized = true`, новая суммаризация сохраняется как `SYSTEM` сообщение с `summary = true`

**SummarizeConfig** — read-only DTO для панели настроек: `tokenThreshold`, `messageCountThreshold`, `overlapMessages`.

### 13. Jira-чаты (JiraChatService)

`JiraChatService` создаёт чаты, предзагруженные контекстом задачи Jira.

**Поток создания (`POST /api/chats/jira`):**
1. Парсинг JIRA URL → issue key
2. Получение содержимого задачи через `JiraService`
3. Создание `ChatTopicEntity` с авто-названием (issue key + summary)
4. Сохранение JIRA-контента как вложения (`.md`, `source_url`)
5. Опционально — получение и сохранение Confluence-страницы
6. Суммаризация всех вложений чата
7. Сохранение приветственного ASSISTANT-сообщения с контекстом задачи

**Обновление (`POST /api/chats/{conversationId}/refresh`):**
- Удаляет старые вложения
- Заново получает данные из Jira
- Пересоздаёт вложения и суммаризацию
- Сообщения чата не трогает

### 14. Библиотека фраз (PhraseController)

Фразы — готовые подсказки над полем ввода в пустом чате. Хранятся в БД (таблица `phrase`) и управляются через админку.

**Публичные эндпоинты** (`/api/phrases`):
- `GET /api/phrases` — только enabled, `ORDER BY category, position`
- `PATCH /api/phrases/{id}/favorite?value=` — переключить избранное

**Админские эндпоинты** (`/api/admin/phrases`):
- `GET /api/admin/phrases?q=` — все фразы (вкл. выключенные), `q` — быстрый поиск по label (ILIKE)
- `POST /api/admin/phrases` — создать (добавляется в конец категории)
- `PUT /api/admin/phrases/{id}` — изменить (смена категории → переезд в конец новой)
- `DELETE /api/admin/phrases/{id}` — удалить
- `PATCH /api/admin/phrases/{id}/favorite?value=` — тоггл избранного
- `PATCH /api/admin/phrases/{id}/enabled?value=` — тоггл enabled
- `PATCH /api/admin/phrases/{id}/move` — переупорядочить

`SecurityConfig` — простая stateless HTTP Basic-аутентификация:

### 18. Поток данных при запросе

```
1. Frontend отправляет POST /api/chats/{conversationId}/runs
   body="текст вопроса", ?model= (опционально), ?clientMsgId= (защита от задвоения)

2. ChatRunService.start()
   ├─ Атомарная заявка на чат (409 если уже идёт генерация)
   ├─ Публикует USER_MESSAGE + RUN_STARTED в ChatEventService
   └─ executor.execute(() -> run())

3. run():
   ├─ resolveModel() — параметр запроса → сохранённая модель чата → дефолтная
   ├─ ChatClient.prompt()
   │    ├─ System prompt (prompt/sys.md)
   │    ├─ ChatMemory (последние 50 сообщений)
   │    ├─ User message
   │    └─ ToolContext (conversationId, collector, userName — явно!)
   └─ spec.stream().chatResponse().subscribe(...)

4. Модель отвечает (потоково)
   ├─ Текст → STREAM → ChatEventService.publish()
   ├─ Вызов инструмента → RecordingToolCallback → ToolInvocationCollector
   │   └─ TOOL_CALL (с resultGist + resultMeta) → ChatEventService.publish()
   └─ Завершение сегмента → TOOL_CALLS → ChatEventService.publish()

5. onComplete:
   ├─ chatMemoryService.saveToolCalls()
   ├─ RUN_DONE → ChatEventService.publish()
   └─ summarizeService.trySummarize() (асинхронно)

6. doFinally (CANCEL/ON_ERROR):
   ├─ persistPartial() — сохраняет накопленный текст с маркером [stopped]/[error]
   └─ RUN_STOPPED/RUN_ERROR → ChatEventService.publish()

7. Frontend: GET /api/chats/{id}/events
   ├─ Replay пропущенных событий (seq > fromSeq)
   └─ Живые события → applyChatEvent() → setChats()
```

### 20. Фича `/file` в композере

В композере чата ([MessageInput.jsx](обзор-чат-системы.md)) можно набрать `/file` и получить fuzzy-поиск по tracked-файлам репозитория. Выбранный файл вставляется как чип (токен `⟦file:PATH⟧`), при отправке разворачивается в fenced code block с содержимым.

#### 20.1 Токены (fileChips.js)

Файл в композере хранится как атомарный токен в плоской строке. Три вида токенов:

| Токен | Назначение | При отправке |
|---|---|---|
| `⟦file:PATH⟧` | Весь файл | Fenced code block с содержимым |
| `⟦file:PATH#FROM-TO⟧` | Диапазон строк (1-based включительно) | Fenced code block с диапазоном |
| `⟦ref:PATH⟧` | Только ссылка | `PATH` (без содержимого) |

**Ключевые функции:**
- `makeToken(path, from, to)` — создать токен «весь файл / диапазон»
- `makeRefToken(path)` — создать токен «только путь»
- `parseToken(token)` — разобрать строку-токен → `{ path, from, to, refOnly }`
- `baseName(path)` — короткое имя файла из пути
- `TOKEN_RE` — глобальный RegExp для поиска всех токенов в строке

#### 20.2 Режимы чипа

- **С содержимым** (`⟦file:PATH⟧`) — при отправке разворачивается в полный fenced code block с языком и содержимым
- **Только ссылка** (`⟦ref:PATH⟧`) — при отправке вставляется только путь (`PATH`)
- **Превью:** клик по чипу открывает полноэкранную модалку (`FileChipPreview`) с содержимым файла и кнопкой переключения режимов (file ↔ ref)

#### 20.3 Кеширование

Содержимое файлов кешируется в `contentCache` (Map) на время жизни сессии. Ключ: `path#from-to`. Функция `fetchContent(path, from, to, signal)` — сначала проверяет кеш, при промахе вызывает `gitApi.getFileContent()`.

#### 20.4 Отправка (expandTokensForSend)

При отправке сообщения `MessageInput.handleSubmit()` вызывает `expandTokensForSend(text)`:
1. Находит все токены через `TOKEN_RE`
2. Для каждого токена параллельно (`Promise.all`) получает содержимое через `fetchContent()`
3. `⟦ref:PATH⟧` → `` `PATH` ``
4. `⟦file:PATH⟧` → fenced code block: `` `PATH`:\n```lang\ncontent\n``` ``
5. Подбирает минимальный fence из бэктиков, не конфликтующий с содержимым
6. Для бинарных файлов: `` `PATH`: [бинарный файл] ``
7. При ошибке загрузки: `` `PATH`: [не удалось прочитать файл] ``
8. Заменяет токены в исходной строке на развёрнутые блоки

#### 20.5 Фронтенд-компоненты

| Компонент | Файл | Размер | Назначение |
|---|---|---|---|
| `MessageInput` | `MessageInput.jsx` | 88 строк | Композер чата: интеграция `FileChipInput`, вызов `expandTokensForSend()` перед отправкой |
| `FileChipInput` | `FileChipInput.jsx` | 504 строки | Rich-композер с чипами: рендерит токены как визуальные чипы, обрабатывает `/file` и `/ref`, управляет дропдауном и превью |
| `FilePickerDropdown` | `FilePickerDropdown.jsx` | ~100 строк | Плавающий дропдаун fuzzy-поиска файлов через `gitApi.searchFiles()` |
| `fileChips` | `fileChips.js` | 103 строки | Модель токенов, парсинг, кеш содержимого (`contentCache`), `expandTokensForSend()` |
| `gitApi` | `gitApi.js` | 30 строк | Обёртка над `/api/git/*`: `searchFiles(q, limit)` и `getFileContent(path, from, to)` |

#### 20.6 Бэкенд (GitController)

`GitController` (`/api/git`) предоставляет read-only эндпоинты для автодополнения `/file`:

- `GET /api/git/files/search?q=&limit=` — fuzzy-поиск tracked-файлов по имени (например, `q=mgi` → `MessageInput`). Возвращает `GitFileNode[] { path, name, type, size }`
- `GET /api/git/files/content?path=&from=&to=` — содержимое файла с опциональным диапазоном строк (1-based включительно). Возвращает `GitFileContent { path, content, binary, sizeBytes, language, totalLines, ... }`

#### 20.7 Поток взаимодействия

```
Пользователь вводит /file в композере
    │
    ▼
FileChipInput — перехватывает /file, показывает FilePickerDropdown
    │
    ▼
FilePickerDropdown — вызывает gitApi.searchFiles(q) → GET /api/git/files/search
    │
    ▼
Пользователь выбирает файл → FileChipInput вставляет токен ⟦file:PATH⟧
    │  (или ⟦ref:PATH⟧ через переключение режима)
    ▼
Пользователь нажимает «Отправить»
    │
    ▼
MessageInput.handleSubmit() → expandTokensForSend(text)
    ├─ fetchContent() → gitApi.getFileContent() → GET /api/git/files/content

    │
```
