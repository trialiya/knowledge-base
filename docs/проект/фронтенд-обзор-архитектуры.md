## 3. useAppNavigation — централизованная навигация

**Единственный владелец URL** во всём приложении. Только этот хук пишет в `window.history` и слушает `popstate`. Никакие другие компоненты историю не трогают.

### Принципы

- Весь URL-стейт живёт в одном объекте `nav`
- Каждый осознанный переход = ровно один `pushState`
- `popstate` просто читает URL обратно в состояние — UI перерисовывается
- `docId` в `nav` строго соответствует URL. «Последний документ» хранится отдельно в `lastKbRef` (вне URL), чтобы `switchView('knowledge')` мог восстановить его даже после «Назад» между чатами (где doc в URL не пишется)

### Состояние `nav`

| Поле | Тип | Описание |
|---|---|---|
| `view` | `'chat' \| 'knowledge'` | Активная вкладка |
| `chatId` | `string \| null` | Последний открытый чат (живёт даже в KB-view) |
| `docId` | `string \| null` | Документ, активный для ТЕКУЩЕЙ записи истории |
| `docTab` | `string` | Вкладка детали документа (summary/content/contents/attachments) |
| `search` | `string` | Поисковый запрос KB |
| `mode` | `string` | Режим поиска (hybrid/semantic/keyword) |

### lastKbRef — память последнего документа KB (вне URL)

Нужна, потому что `docId` в `nav` обнуляется при «Назад» на chat-запись (в chat-URL doc не пишется). `lastKbRef` помнит последний реально открытый документ и его вкладку, чтобы `switchView('knowledge')` мог его восстановить. Обновляется при каждом `openDoc` и при изменении `nav.docId` (когда он не null).

### URL-схема

```
?view=chat|knowledge&chat=<id>&doc=<id>&tab=<tab>&search=<q>&mode=<m>
```

- `view=chat` → в URL пишутся только `view` и `chat` (doc/tab/search/mode — не пишутся)
- `view=knowledge` → `view`, `chat`, и взаимоисключающие `doc` или `search`
- `chatId` сохраняется в обоих view, чтобы возврат в чат помнил активный чат

### Публичные методы

| Метод | Действие |
|---|---|
| `switchView(view)` | Переключить верхнюю вкладку. Для KB восстанавливает последний документ из `lastKbRef`, если для текущей записи истории документ/поиск не активны |
| `openDoc(docId, docTab?)` | Открыть документ в KB (из чата, doc-ссылки, дерева). Сбрасывает search. Обновляет `lastKbRef` |
| `setDocTab(tab)` | Сменить вкладку детали документа |
| `setSearch(query, mode)` | Запустить поиск в KB (сбрасывает выбранный документ) |
| `openChat(chatId)` | Открыть/сменить активный чат |

### Механика записи URL

1. При изменении `nav` → `buildSearch(nav)` строит query-строку
2. Если query не изменился — `pushState` не вызывается (дедупликация)
3. Если изменение пришло из `popstate` — `pushState` также пропускается (URL уже актуален)
4. При старте: если текущий URL не содержит явного `view`, `replaceState` добавляет его

### popstate

Обработчик читает URL → обновляет `nav`:
- `view` из URL, иначе inferred (doc/search → knowledge, иначе chat)
- `chatId`/`docId`: из URL, иначе сохраняется прежнее значение (память последнего)
- Флаг `fromPopRef` предотвращает повторную запись URL

## 4. App.js — корень приложения

**Два таба, оба всегда смонтированы** (скрытие через CSS, не размонтирование). Это сохраняет:
- Позицию скролла
- Загруженные данные
- Состояние стриминга

### Навигация между табами

Вся навигационная логика вынесена в `useAppNavigation`. App.js только пробрасывает пропсы:

```
useAppNavigation() → { nav, switchView, openDoc, setDocTab, setSearch, openChat }
```

| Проп | ChatWindow | KnowledgeBase |
|---|---|---|
| `isActive` | ✅ | ✅ |
| `activeChatId` / `docId` | `nav.chatId` | `nav.docId` |
| `onSelectChat` / `onOpenDoc` | `openChat` | `openDoc` |
| `onNavigateToDoc` | `openDoc` | — |
| `onTabChange` | — | `setDocTab` |
| `onSearch` | — | `setSearch` |
| `onNavigateToChat` | — | `openChat` |

**Больше нет:** `app:navigate-doc` и `app:navigate-chat` событий, `lastDocIdRef`, `navigateToDoc`/`navigateToChat` в App.js, `popstate` в App.js.

### Глобальная строка поиска (в хедере вкладок)

Поисковая строка вынесена из KB-хедера в глобальный хедер вкладок (`app-tabs`). Видна всегда, независимо от активной вкладки.

**Структура хедера (3 зоны):**
- **Левая** (`app-tabs__left`, flex:1) — кнопки вкладок «💬 Чаты» / «📚 База знаний» (i18n: `nav.chats`, `nav.knowledgeBase`)
- **Центр** (`app-search-row`, flex:0 1 480px) — поисковый input + select режима (hybrid/semantic/keyword). Enter запускает поиск, смена режима при активном запросе — мгновенно перезапускает
- **Правая** (`app-tabs__right`, flex:1) — `LanguageSwitcher` (переключатель RU/EN) + кнопка Refresh (только при `view=knowledge` и открытом документе)

**Синхронизация с URL:** `searchText` и `searchMode` синхронизируются с `nav.search`/`nav.mode` через `useEffect`. При вводе — локальное состояние; при submit — `setSearch()` → `useAppNavigation` → URL.

### LanguageSwitcher — переключатель языка

Компонент в правой части хедера вкладок. Кнопка с иконкой глобуса (🌐) + выпадающее меню RU/EN. При выборе языка вызывает `i18n.changeLanguage(lng)`. Меню закрывается по Esc и клику снаружи. Стили в `common/languageSwitcher.css`.

### Кнопка Refresh

Видна только при `view=knowledge` и `nav.docId !== null`. Инкрементит `refreshTick` → `KnowledgeBase` через `useEffect` вызывает `handleRefresh()`. Статус `refreshing` поднимается колбэком `onRefreshingChange` для спиннера и блокировки кнопки.

### Защита от потери несохранённых изменений

При попытке переключиться с KB на чат, если в редакторе есть несохранённые изменения (`isEditorDirty()`), показывается `ConfirmModal`:

- **Заголовок:** «Несохранённые изменения» (i18n: `unsaved.title`)
- **Сообщение:** «В редакторе базы знаний есть несохранённые изменения. Перейти в чат? Они останутся в редакторе, но не будут сохранены.» (i18n: `unsaved.message`)
- **«Перейти в чат»** — выполняет `switchView('chat')`
- **«Остаться»** — закрывает модалку, пользователь остаётся в KB

Обе панели смонтированы всегда (скрытие через CSS), поэтому правки **не теряются** при переключении — это просто предупреждение.

## 5. chatPanel — компоненты чата

### ChatWindow — ключевые механики

**Состояние:**
- `chats` — список чатов (id, title, messages, createdAt, jiraUrl, notFound, loadError, model)
- `activeChatId` — источник правды: проп `propActiveChatId` из `useAppNavigation`. Локальные выборы поднимаются наверх через `onSelectChat` и возвращаются как проп
- `isLoading` — флаг стриминга
- `attachPanelOpen` / `attachCount` — панель вложений
- `hasMore` / `oldestCursor` / `pendingLeadingMetas` — состояние пагинации

**Создание чата (черновик):** новый чат создаётся с `id = 'new'` (черновик) и `messages: []`. Реальный UUID рождается при отправке первого сообщения. Черновик не показывается в сайдбаре, пока в нём нет сообщений. При «удалении» черновика (кнопка корзины) — очищается только поле ввода, сам черновик остаётся.

**Навигация (централизованная):**
- `selectChat(id)` — устанавливает `activeChatId` локально + вызывает `onSelectChat(id)` (поднимает в `useAppNavigation`). URL пишет только навигация
- `propActiveChatId` — когда меняется (popstate, восстановление из URL), `useEffect` подхватывает и обновляет локальный `activeChatId`
- **Больше нет:** `getUrlState`, `setChatUrlState`, `lastUrlChatIdRef`, слушатель `app:navigate-chat`, эффект синхронизации URL

**Обработка устаревших ссылок на чаты:**
- `initialPropChatIdRef` — запоминает `propActiveChatId` на момент монтирования, чтобы отличить «ссылку из URL» от «id из localStorage»
- Если `?chat=<id>` явно задан в URL, а чата не существует — показывается ошибка «не найдено»
- Если id пришёл из localStorage (устаревший) — молча переключаемся на первый существующий чат

**Выбор модели:**
- `ModelSelector` — кастомный дропдаун в шапке чата (моноширинный шрифт, стрелка вплотную к тексту, клавиатурная навигация Arrow/Home/End/Escape)
- `lastModelRef` — localStorage для запоминания последней выбранной модели
- `makeDraft()` — черновик с моделью из `lastModelRef`
- `handleModelChange(modelId)` — оптимистичное обновление UI + `PUT /api/chats/{id}/model`
- Конфиг моделей загружается при монтировании: `GET /api/chats/models`
- i18n: `model.aria` (label), `model.default` (пометка «по умолчанию»)

**Стриминг SSE:**
1. POST `/api/chats/{id}/messages/stream` с телом сообщения
2. Чтение `response.body.getReader()` — построчная обработка `data:` строк
3. Типы событий:
   - `data: {"message":"текст"}` — текстовый чанк
   - `data: {"toolCall":{...}}` — вызов инструмента (с `resultMeta`)
   - `data: {"toolCalls":[...]}` — финальный список вызовов (с `resultMeta`)
   - `data: {"finishReason":"TOOL_CALLS"}` — сегмент завершён, начинается новый
   - `data: {"finishReason":"DONE"}` — полное завершение
   - `data: [DONE]` — альтернативный сигнал завершения
4. **Сегментация:** при `TOOL_CALLS` создаётся новое AI-сообщение (если в текущем есть текст)
5. **Abort:** через `AbortController` — кнопка «Стоп»
6. **Обработка ошибок стрима:** при ошибке сохраняется частичный ответ + tool calls. Если что-то успело прийти — дописывается `[⚠ Ответ прерван из-за ошибки на сервере]`, иначе — стандартное сообщение об ошибке

**Загрузка чатов:**
- `GET /api/chats` → список чатов
- Если `activeChatId` из URL не найден — создаётся заглушка, показывается ошибка
- `failedChatIdsRef` защищает от повторных запросов к несуществующим чатам

**Загрузка сообщений (пагинация):**
- `PAGE_SIZE = 20`
- `loadMessages(chatId)` — два параллельных запроса:
  - `GET /api/chats/{id}?includeMessages=false` — метаданные чата (model и др.)
  - `GET /api/chats/{id}/messages?limit=20` — последние сообщения
- `transformPage(messages)` — превращает сырые сообщения в пузыри, прикрепляя `toolInvocationMetas` к предыдущему AI-сообщению. Если ассистент остался в более старой странице — metas уходят в `leadingMetas`
- `attachLeadingMetas()` — прицепляет «висячие» metas при догрузке
- `loadOlderMessages()` — догрузка старых сообщений при прокрутке вверх: `GET /api/chats/{id}/messages?beforeCreatedAt=...&beforeId=...&limit=20`
- `handleLoadOlder` — колбэк для `MessageList`
- **Обработка SYSTEM-сообщений:** SYSTEM-сообщения не рендерятся как пузыри. Их `toolInvocationMetas` с `resultMeta` прикрепляются к предыдущему AI-сообщению

**Переименование:**
- `PUT /api/chats/{id}/topic` (тело — plain text)
- Фоновое обновление темы после ответа AI: `GET /api/chats/{id}?...`
- **Сохранение createdAt:** при обновлении темы `createdAt` больше не затирается (`chat.createdAt ?? data.createdAt ?? null`)

### useKnowledgeBase — центральный хук

**Пропсы (из `useAppNavigation` через `KnowledgeBase.jsx`):**

| Проп | Тип | Описание |
|---|---|---|
| `docId` | `string \| null` | Какой документ показать |
| `docTab` | `string` | Вкладка детали (summary/content/…) |
| `search` | `string` | Поисковый запрос |
| `mode` | `string` | Режим поиска |
| `onOpenDoc(id, tab?)` | callback | KB сообщает навигации: выбран документ |
| `onSearch(query, mode)` | callback | KB сообщает навигации: запущен поиск |
| `onTabChange(tab)` | callback | KB сообщает навигации: сменилась вкладка детали |

**Хук НЕ пишет URL и НЕ слушает popstate** — этим владеет `useAppNavigation`.

**Состояние:**
- `tree` — дерево узлов
- `selectedNode` — выбранный узел
- `activeTab` — вкладка (summary/content/contents/attachments)
- `searchQuery`, `searchMode`, `searchResults`
- `showAddModal`, `notFoundDocId`, `docLoadError`, `saveError`
- `refreshing`, `moveConfirm`, `discardConfirm`, `deleteConfirm`

**Загрузка дерева:**
- `loadTree()` — корневой уровень через `GET /api/documents/children`
- `handleLoadChildren(parentId, page)` — ленивая загрузка при раскрытии папки
- `refreshScope(parentId)` — после CRUD-операций

**materializePathTo(targetId, baseTree):**
Общий метод для прямой ссылки и refresh. Загружает полный список детей каждого предка (FULL_PAGE), маркирует их `_openOnLoad`, прошивает растущее дерево через локальную переменную и `setTree`. Возвращает финальное дерево. Заменил две почти идентичные копии (в `navigateToDocById` и `handleRefresh`).

**Навигация (централизованная):**
- `selectNode(nodeOrId)` — выбор узла. Всегда делает `fetchById` для получения полного `description`. Для SearchResult (без поля `type`) — также `fetchById` перед отображением. Вызывает `onOpenDoc(id, 'summary')` с `notify: true`
- `navigateToDocById(docId, opts)` — загрузка цепочки ancestors → раскрытие → выбор. Используется при восстановлении из URL (с `notify: false`, чтобы избежать петли)
- `applySelectNode(node, {notify, tab})` — `notify: true` для пользовательских действий (уведомляет навигацию), `notify: false` когда выбор пришёл ИЗ навигации (prop `docId`). Параметр `tab` позволяет сохранять вкладку из URL при back/forward (раньше был жёсткий `'summary'`)
- **Больше нет:** `getUrlState`, `setKBUrlState`, `popstate`-слушателя, `app:navigate-doc`-слушателя, инициализации из URL

**Реакция на навигацию (props):**
- Единственный `useEffect` на `navDocId`, `navSearch`, `navDocTab`, `navMode`
- `navDocId` → `navigateToDocById(docId, {notify: false, tab})` (не уведомляет навигацию обратно)
- Тот же документ, но другая вкладка (back на запись истории) → только `setActiveTab(tab)`, без перезагрузки
- `navSearch` → `performSearch(query, mode)`
- `lastNavDocRef` / `lastNavSearchRef` — защита от повторной обработки того же значения

**Защита от дублирующихся запросов:**
- `didInitRef` — предотвращает двойной вызов инициализации в StrictMode
- `inflightChildrenRef` — дедупликация параллельных запросов children (tree expand + folder detail)

**Защита от потери несохранённых изменений (editorDirtyStore):**

`useKnowledgeBase` оборачивает все навигационные действия в guard-функцию:

- `guard(action)` — если `isEditorDirty()`, откладывает `action` в `pendingActionRef` и показывает `discardConfirm`-модалку. Иначе выполняет сразу
- `handleDiscardConfirm` — сбрасывает dirty-флаги через `clearEditorDirty()`, выполняет отложенное действие
- `handleDiscardCancel` — сбрасывает отложенное действие, закрывает модалку

**Обёрнутые (guarded) методы:**
- `guardedSelectNode` — выбор узла (дерево, поиск, doc-ссылка)
- `guardedChangeTab` — смена вкладки детали
- `guardedSearch` — запуск поиска
- `guardedRefresh` — обновление документа (перезагружает с API → правки теряются)

Также хук слушает `beforeunload` — если редактор dirty, браузер покажет стандартное предупреждение при закрытии вкладки.

**CRUD:**
- Создание: `POST /api/documents`

- Удаление: `DELETE /api/documents/{id}` с подтверждением через `ConfirmModal`

### editorDirtyStore — модуль-уровневое хранилище dirty-флагов

**Проблема:** `DocumentDetail` рендерится с `key={node.id}`, поэтому при смене документа редактор размонтируется — его внутренний `dirty`-стейт теряется до того, как навигация принимает решение.

**Решение:** модуль-уровневый `Set`-реестр (как кэш в `useDocPreview`):

| Функция | Назначение |
|---|---|
| `setEditorDirty(id, value)` | Зарегистрировать/снять dirty-флаг экземпляра редактора |
| `isEditorDirty()` | `true` если хотя бы один редактор dirty |
| `clearEditorDirty()` | Сбросить все флаги (после подтверждения discarding) |
| `subscribeEditorDirty(cb)` | Подписка на изменения (возвращает unsubscribe) |

**Ключевание по instance id** (а не один булев флаг): инлайн-редактор и полноэкранный «развернуть» могут быть смонтированы одновременно. Чистый экземпляр не должен затирать dirty-статус другого.

**Интеграция в MarkdownEditor:**
- При монтировании: `setEditorDirty(dirtyId, dirty)` в `useEffect` на `[dirty, previewOnly]`
- При размонтировании: cleanup `setEditorDirty(dirtyId, false)` — чтобы «мёртвый» редактор не блокировал навигацию
- `previewOnly`-экземпляры (SummarySection) никогда не регистрируются

### TreeNode — узел дерева

- **Раскрытие:** `toggleOpenVisual` вместо `toggleOpen` — не делает отдельный fetch, полагается на shared loader из `useFolderChildren`. Это устраняет дублирующиеся запросы при раскрытии папки.
- **«Load more»:** кнопка внизу списка детей, если `totalElements > children.length`. Текст через `t('tree.loadMore', { count: remaining })`
- **Drag & Drop:** module-level `dragState` (обход ограничений браузера)
  - Системные узлы не перетаскиваются
  - Drop-позиции: before, after, inside (для папок)
  - Визуальные индикаторы через CSS-классы
  - Drag handle с i18n-тултипом `t('tree.dragToReorder')`
- **Авто-открытие:** `_openOnLoad` для навигации по прямой ссылке
- **Авто-раскрытие:** если выбранный узел — потомок
- **i18n:** кнопка удаления `t('tree.delete')`, системный бейдж `t('detail.systemBadge')`

### useFolderChildren — shared loader

- Данные детей принадлежат shared tree, а не хуку. Хук **выводит** `children` из `node.children`, а не держит второй копии в состоянии
- `direct`-состояние только для fallback-пути без loader'а
- Проверяет `treeCacheIsComplete` — если дети уже загружены, не делает повторный запрос
- Использует общий `loadChildren` вместо прямого `api.fetchChildren`

**Загрузка данных (ленивая):**
- `api.fetchHistory(id)` → `GET /api/documents/{id}/history` — возвращает `DocumentHistoryShort[]` (без `description`, только метаданные)
- При выборе версии: `api.fetchHistoryVersion(id, version)` → `GET /api/documents/{docId}/history/{version}` — возвращает полный `DocumentHistory` с `description`
- **Кэш описаний:** `descCache` (Map) — загруженные описания не запрашиваются повторно
- **Дедупликация запросов:** `inFlight` — `Map<version, Promise<string>>`. `ensureLoaded` возвращает Promise, что позволяет и предпросмотру/diff, и «Восстановить» использовать один путь
- **Предзагрузка:** `ensureLoaded` загружает базовую версию (текущую) и изменённую (выбранную) — остальные версии не грузятся

**initialVersion — автонаведение на версию:**
Новый проп. Если передан (например, из `DocChangeBlock` в чате), `HistoryModal` автоматически находит запись с этой `descriptionVersion` и показывает diff с предыдущей версией. Если это старейшая запись — показывает её саму (без diff).

**Структура панели (две колонки):**
- **Левая колонка** — список версий (хронологически, сверху новая). Каждая версия показывает дату, автора и сниппет описания. Текущая версия помечена бейджем «текущая». Активная версия подсвечена
- **Правая колонка** — предпросмотр выбранной версии (рендеринг Markdown через `ReactMarkdown` + `remarkGfm`)

**Diff-представление:**
- При выборе версии (не текущей) — кнопка «Сравнить с текущей»
- Diff вычисляется библиотекой `diff` (^9.0.0): `diff.diffWords(oldText, newText)`
- Добавленные фрагменты подсвечиваются зелёным фоном (`.history-diff-added`), удалённые — красным с зачёркиванием (`.history-diff-removed`)

**Восстановление версии:**
- Кнопка «Восстановить эту версию» (только для не-текущих версий)
- Использует `ensureLoaded` (переиспользует кэш или догружает на лету)
- При клике: `PUT /api/documents/{id}` с `description` выбранной версии
- После успешного восстановления: модалка закрывается, документ обновляется (refresh)

**Закрытие:** Esc (через `useEscape`) или клик по оверлею.

### MoveConfirmModal — подтверждение перемещения

Модальное окно подтверждения при смене родительской папки через drag & drop. Интернационализировано через `t('move.*')`.

- **Заголовок:** «Переместить документ» или «Переместить папку»
- **Сообщение:** «Переместить [название] в папку [родитель]?»
- **Кнопки:** «Переместить» / «Отмена»

### DocLinkTooltip — ховер-превью для KB-ссылок

Обёртка для ссылок `/?doc=N` в рендеренном Markdown. При наведении показывает портал-тултип с превью документа. Интернационализировано через `t('docLink.*')`, даты форматируются с учётом локали.

- **Кэширование:** `useDocPreview` — сначала ищет в tree (мгновенно), затем в module-level cache, затем fetch
- **Дедупликация:** параллельные запросы к одному документу ждут один и тот же промис
- **Навигация:** через проп `onNavigate` (в KB это `selectNode`, принимающий id). `selectNode → fetchFullAndSelect(id, {notify:true})` → документ грузится, выбирается, и `useAppNavigation` обновляет URL. Фолбэк на `window.dispatchEvent('app:navigate-doc')` оставлен для обратной совместимости
- **Задержка:** 180ms до показа, 200ms до скрытия
- **Позиционирование:** фиксированное, авто-флип (сверху/снизу), не выходит за границы экрана
- **Содержимое тултипа:** заголовок + тип, AI-summary (если есть) или сниппет description, дата, кнопка «Открыть →»

### useDocPreview — кэш превью документов

Модуль-уровневый кэш (живёт всю жизнь страницы):

1. **Tree lookup:** если узел есть в дереве с загруженным description → мгновенный ответ
2. **Module cache:** `Map<id, node|'loading'|'error'>` — повторные ховеры на ту же ссылку мгновенны
3. **Inflight dedup:** если запрос уже в полёте → подписывается на результат через `listeners`
4. **Fresh fetch:** `api.fetchById(id)` → сохраняет в кэш → уведомляет подписчиков
5. **Cancellation-aware:** если `id`/`enabled` меняются во время запроса, cleanup выставляет `cancelled`, и сеттеры не вызываются для устаревшего id (но кэш всё равно заполняется через `notify`)

### useAtMention — автокомплит @mention

Хук, подключаемый к textarea через ref:

- **Детект:** regex `(?:^|[\\s\\n])@([^\\s@]*)$` на тексте до курсора
- **Дебаунс:** 200ms перед запросом
- **Поиск:** `api.searchByName(q, MAX_RESULTS, controller.signal)` — использует `AbortController` для отмены устаревших запросов
- **Позиционирование:** mirror-div для точного положения каретки
- **Клавиатура:** ↑↓ — навигация, Enter — выбрать, Esc — закрыть
- **Вставка:** заменяет `@query` на `[Название](/?doc=ID)`, курсор после ссылки
- **dismiss/select** объявлены до `handleKeyDown` (исправление stale-closure)

### Поиск (SearchResults)

- Карточки результатов: заголовок, дата, хлебные крошки, сниппет. Интернационализировано через `t('search.*')`, даты форматируются с учётом локали
- Хлебные крошки приходят с бэкенда в `parentList` (батчевый CTE-запрос), а не строятся на фронте через `findPath()`
- Заголовок — `<a href>` с поддержкой средней кнопки мыши и Ctrl+клик

### api.js — API-обёртки

Унифицированный контракт: read-методы проверяют `response.ok` и выбрасывают типизированную ошибку (`err.status`), write-методы возвращают сырой Response.

| Метод | Эндпоинт |
|---|---|
| `fetchTree()` | `GET /api/documents/tree` |
| `fetchChildren(parentId, page, size)` | `GET /api/documents/children` |
| `fetchById(id)` | `GET /api/documents/{id}` |
| `fetchAncestors(id)` | `GET /api/documents/{id}/ancestors` |
| `fetchHistory(id)` | `GET /api/documents/{id}/history` → `DocumentHistoryShort[]` |
| `fetchHistoryVersion(id, version)` | `GET /api/documents/{docId}/history/{version}` → `DocumentHistory` |
| `searchDocuments(query, mode, ...)` | `GET /api/documents/search` |
| `searchByName(name, limit, signal)` | `GET /api/documents/search-by-name` (с AbortSignal) |
| `createDocument(body)` | `POST /api/documents` |
| `updateDocument(id, body)` | `PUT /api/documents/{id}` |
| `deleteDocument(id)` | `DELETE /api/documents/{id}` |
| `reorder(body)` | `PATCH /api/documents/reorder` |
| `summarize(id)` | `POST /api/documents/{id}/summarize` |

Вспомогательные функции: `ensureOk(r, label)` — выбрасывает типизированную ошибку для не-OK ответов; `json(r, label)` — `ensureOk` + `.json()`.

### treeOps — чистые операции над деревом

| Функция | Назначение |
|---|---|
| `cloneTree(tree)` | Структурный deep clone (единый источник идиомы клонирования) |
| `getSiblings(tree, parentId)` | Дети заданного родителя |
| `applyReorder(tree, dropInfo)` | Применить перемещение (возвращает новый массив) |
| `updateNodeInTree(tree, id, updates)` | Обновить узел по id |
| `spliceChildren(clone, parentId, paged, opts)` | Вставить страницу детей в родителя (мутирует clone, возвращает parent) |
| `applyChildren(tree, parentId, paged, opts)` | Иммутабельная версия: clone → splice → mark `_openOnLoad` → возвращает новый tree |
| `parentTitle(tree, parentId, t?)` | Имя родителя (null → «Корневой уровень»). Опциональный `t` для i18n |
| `isParentChange(dropInfo)` | Изменился ли родитель при drop |

### Отправка сообщения в чат
```
MessageInput (Enter)
  → ChatWindow.handleSendMessage(text)
    → setChats (добавляет user + ai сообщения)
    → fetch POST /api/chats/{id}/messages/stream (SSE)
      → читаем data: чанки
        → textChanged: обновляем aiMessageTextRef + setChats
        → toolCall: обновляем toolCallsRef + setChats (с resultMeta)
        → TOOL_CALLS: финализируем сегмент, создаём новый
        → DONE: финализируем всё, fetchAndUpdateTitle
      → onError: сохраняем частичный ответ + tool calls
        → если что-то пришло: дописываем [⚠ Ответ прерван из-за ошибки на сервере]
        → иначе: стандартное сообщение об ошибке
```

### Загрузка истории чата (с пагинацией и DocChangeBlock)
```

  → параллельно:
    → GET /api/chats/{id}?includeMessages=false (метаданные: model и др.)

  → transformPage(messages)
    → SYSTEM-сообщения с toolInvocationMetas:
      → не рендерятся как пузыри

        → resultMeta попадает в toolCalls
    → Message: DocChangeBlock видит toolCalls с resultMeta

Догрузка старых сообщений:
  → MessageList: scrollTop < 120px → maybeLoadMore()

```

```
    → Portal-тултип: заголовок, summary/сниппет, дата, «Открыть»
  → onClick: onNavigate(id) → selectNode(id)
    → guard → fetchFullAndSelect(id, {notify: true})
      → onOpenDoc(id) → useAppNavigation.openDoc(id)
```

### Смена вкладки чата
```
ChatList click / popstate
  → selectChat(id)
    → setActiveChatId(id) локально
    → onSelectChat(id) → useAppNavigation.openChat(id)
      → setNav({view:'chat', chatId:id})
      → pushState в историю
```

### Глобальный поиск
```
App.js: search input → Enter
  → submitSearch() → setSearch(query, mode)
    → useAppNavigation.setSearch(query, mode)
      → setNav({view:'knowledge', search:query, mode, docId:null})
      → pushState в историю
    → useKnowledgeBase: useEffect на navSearch
      → performSearch(query, mode)
```

### Refresh документа
```
App.js: кнопка Refresh (только при view=knowledge + docId)
  → setRefreshTick(tick + 1)
  → KnowledgeBase: useEffect на refreshSignal
    → handleRefresh() (guarded — с проверкой dirty)
      → loadTree → materializePathTo → fetchFullAndSelect
```

### История изменений описания (ленивая загрузка)
```
DocumentDetail / FolderDetail / MarkdownEditor → кнопка «История» (IconHistory)
  → HistoryModal: api.fetchHistory(id) → GET /api/documents/{id}/history
    → DocumentHistoryShort[] (без description)
    → список версий (левая колонка)
    → выбор версии → ensureLoaded(version) → Promise<string>
      → api.fetchHistoryVersion(id, version) если не в кэше
      → GET /api/documents/{docId}/history/{version} → DocumentHistory (с description)
      → кэш descCache + inFlight (Map) для дедупликации
      → предпросмотр Markdown (правая колонка)
    → «Сравнить с текущей» → ensureLoaded(базовая + изменённая) → diff.diffWords(oldText, newText)
      → зелёный фон (добавлено), красный зачёркнутый (удалено)
    → «Восстановить эту версию» → ensureLoaded(version) → onRestore(text)
      → PUT /api/documents/{id} (description выбранной версии)
      → модалка закрывается, документ обновляется
```

### DocChangeBlock — карточки изменений в чате
```
ChatWindow (стрим или загрузка истории)
  → toolCalls с resultMeta (createDocument/updateDocument)
  → Message: DocChangeBlock
    → useMemo: группировка по id, макс. descriptionVersion
    → карточки: 📄 название, «создан/обновлён», v{N}, «Посмотреть изменения ›»
    → клик → HistoryModal с initialVersion
      → автонаведение на diff выбранной версии с предыдущей
```
