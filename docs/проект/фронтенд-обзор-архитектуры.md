## 3. useAppNavigation — централизованная навигация

**Единственный владелец URL** во всём приложении. Только этот хук пишет в `window.history` и слушает `popstate`. Никакие другие компоненты историю не трогают.

### Принципы

- Весь URL-стейт живёт в одном объекте `nav`
- Каждый осознанный переход = ровно один `pushState`
- `popstate` просто читает URL обратно в состояние — UI перерисовывается
- `docId` хранится в памяти всегда (даже когда `view === 'chat'`), чтобы клик по вкладке «База знаний» восстанавливал последний документ. В URL `doc` попадает только при `view === 'knowledge'`

### Состояние `nav`

| Поле | Тип | Описание |
|---|---|---|
| `view` | `'chat' \| 'knowledge'` | Активная вкладка |
| `chatId` | `string \| null` | Последний открытый чат (живёт даже в KB-view) |
| `docId` | `string \| null` | Последний открытый документ (живёт даже в chat-view) |
| `docTab` | `string` | Вкладка детали документа (summary/content/contents/attachments) |
| `search` | `string` | Поисковый запрос KB |
| `mode` | `string` | Режим поиска (hybrid/semantic/keyword) |

### URL-схема

```
?view=chat|knowledge&chat=<id>&doc=<id>&tab=<tab>&search=<q>&mode=<m>
```

- `view=chat` → в URL пишутся только `view` и `chat` (doc/tab/search/mode — не пишутся)
- `view=knowledge` → `view`, `chat`, и взаимоисключающие `doc` или `search`
- `chatId` сохраняется в обоих view, чтобы возврат в чат помнил активный чат

### Публичные методы

| Метод | Действие |
|---|---|
| `switchView(view)` | Переключить верхнюю вкладку. Для KB восстанавливает последний документ из памяти |
| `openDoc(docId, docTab?)` | Открыть документ в KB (из чата, doc-ссылки, дерева). Сбрасывает search |
| `setDocTab(tab)` | Сменить вкладку детали документа |
| `setSearch(query, mode)` | Запустить поиск в KB (сбрасывает выбранный документ) |
| `openChat(chatId)` | Открыть/сменить активный чат |

### Механика записи URL

1. При изменении `nav` → `buildSearch(nav)` строит query-строку
2. Если query не изменился — `pushState` не вызывается (дедупликация)
3. Если изменение пришло из `popstate` — `pushState` также пропускается (URL уже актуален)
4. При старте: если текущий URL не содержит явного `view`, `replaceState` добавляет его

### popstate

Обработчик читает URL → обновляет `nav`:
- `view` из URL, иначе inferred (doc/search → knowledge, иначе chat)
- `chatId`/`docId`: из URL, иначе сохраняется прежнее значение (память последнего)
- Флаг `fromPopRef` предотвращает повторную запись URL

## 4. App.js — корень приложения

**Два таба, оба всегда смонтированы** (скрытие через CSS, не размонтирование). Это сохраняет:
- Позицию скролла
- Загруженные данные
- Состояние стриминга

### Навигация между табами

Вся навигационная логика вынесена в `useAppNavigation`. App.js только пробрасывает пропсы:

```
useAppNavigation() → { nav, switchView, openDoc, setDocTab, setSearch, openChat }
```

| Проп | ChatWindow | KnowledgeBase |
|---|---|---|
| `isActive` | ✅ | ✅ |
| `activeChatId` / `docId` | `nav.chatId` | `nav.docId` |
| `onSelectChat` / `onOpenDoc` | `openChat` | `openDoc` |
| `onNavigateToDoc` | `openDoc` | — |
| `onTabChange` | — | `setDocTab` |
| `onSearch` | — | `setSearch` |
| `onNavigateToChat` | — | `openChat` |

**Больше нет:** `app:navigate-doc` и `app:navigate-chat` событий, `lastDocIdRef`, `navigateToDoc`/`navigateToChat` в App.js, `popstate` в App.js.

### Глобальная строка поиска (в хедере вкладок)

Поисковая строка вынесена из KB-хедера в глобальный хедер вкладок (`app-tabs`). Видна всегда, независимо от активной вкладки.

**Структура хедера (3 зоны):**
- **Левая** (`app-tabs__left`, flex:1) — кнопки вкладок «💬 Чаты» / «📚 База знаний»
- **Центр** (`app-search-row`, flex:0 1 480px) — поисковый input + select режима (hybrid/semantic/keyword). Enter запускает поиск, смена режима при активном запросе — мгновенно перезапускает
- **Правая** (`app-tabs__right`, flex:1) — кнопка Refresh (только при `view=knowledge` и открытом документе)

**Синхронизация с URL:** `searchText` и `searchMode` синхронизируются с `nav.search`/`nav.mode` через `useEffect`. При вводе — локальное состояние; при submit — `setSearch()` → `useAppNavigation` → URL.

### Кнопка Refresh

Видна только при `view=knowledge` и `nav.docId !== null`. Инкрементит `refreshTick` → `KnowledgeBase` через `useEffect` вызывает `handleRefresh()`. Статус `refreshing` поднимается колбэком `onRefreshingChange` для спиннера и блокировки кнопки.

### Защита от потери несохранённых изменений

При попытке переключиться с KB на чат, если в редакторе есть несохранённые изменения (`isEditorDirty()`), показывается `ConfirmModal`:

- **Заголовок:** «Несохранённые изменения»
- **Сообщение:** «В редакторе базы знаний есть несохранённые изменения. Перейти в чат? Они останутся в редакторе, но не будут сохранены.»
- **«Перейти в чат»** — выполняет `switchView('chat')`
- **«Остаться»** — закрывает модалку, пользователь остаётся в KB

Обе панели смонтированы всегда (скрытие через CSS), поэтому правки **не теряются** при переключении — это просто предупреждение.

### ChatWindow — ключевые механики

**Состояние:**
- `chats` — список чатов (id, title, messages, createdAt, jiraUrl, notFound, loadError)
- `activeChatId` — источник правды: проп `propActiveChatId` из `useAppNavigation`. Локальные выборы поднимаются наверх через `onSelectChat` и возвращаются как проп
- `isLoading` — флаг стриминга
- `attachPanelOpen` / `attachCount` — панель вложений

**Навигация (централизованная):**
- `selectChat(id)` — устанавливает `activeChatId` локально + вызывает `onSelectChat(id)` (поднимает в `useAppNavigation`). URL пишет только навигация
- `propActiveChatId` — когда меняется (popstate, восстановление из URL), `useEffect` подхватывает и обновляет локальный `activeChatId`
- **Больше нет:** `getUrlState`, `setChatUrlState`, `lastUrlChatIdRef`, слушатель `app:navigate-chat`, эффект синхронизации URL

**Стриминг SSE:**
1. POST `/api/chat/stream?conversationId=...` с телом сообщения
2. Чтение `response.body.getReader()` — построчная обработка `data:` строк
3. Типы событий:
   - `data: {"message":"текст"}` — текстовый чанк
   - `data: {"toolCall":{...}}` — вызов инструмента
   - `data: {"toolCalls":[...]}` — финальный список вызовов
   - `data: {"finishReason":"TOOL_CALLS"}` — сегмент завершён, начинается новый
   - `data: {"finishReason":"DONE"}` — полное завершение
   - `data: [DONE]` — альтернативный сигнал завершения
4. **Сегментация:** при `TOOL_CALLS` создаётся новое AI-сообщение (если в текущем есть текст)
5. **Abort:** через `AbortController` — кнопка «Стоп»

**Загрузка чатов:**
- `GET /api/chat` → список чатов
- Если `activeChatId` из URL не найден — создаётся заглушка, показывается ошибка
- `failedChatIdsRef` защищает от повторных запросов к несуществующим чатам

**Загрузка сообщений:**
- `GET /api/chat/chat?conversationId=...`
- При 404 — чат помечается `notFound: true`
- При других ошибках — `loadError: status`

**Переименование:**
- `POST /api/chat/topic?conversationId=...` (тело — plain text)
- Фоновое обновление темы после ответа AI: `GET /api/chat/chat?...`

**JIRA-чаты:**
- Создание: `POST /api/chat/jira` → новый чат в списке
- Панель вложений: `JiraAttachmentPanel` вместо `AttachmentPanel`
- Refresh: `POST /api/chat/jira/{id}/refresh`

**Защита от гонки URL:** `ChatWindow` принимает проп `isActive` — синхронизирует URL только когда вкладка чата активна. Это предотвращает перезапись `view=knowledge` при монтировании.

**Унификация хедера:** кнопка удаления чата теперь использует иконку `IconTrash` вместо текста «Удалить» — в стиле KB.

### ChatDocLink — превью KB-ссылок в чате

Аналог `DocLinkTooltip` для сообщений чата. Отличия:
- Дерева KB в чате нет → `useDocPreview` грузит документ через `api.fetchById` (tree=[])
- Навигация через проп `onNavigateToDoc` (= `openDoc` из `useAppNavigation`) — переключает вкладку на «База знаний» и пишет URL централизованно
- **Больше нет:** фолбэка на `window.dispatchEvent('app:navigate-doc')`
- Внешние ссылки рендерятся как обычный `<a target="_blank">`
- Использует те же CSS-классы `.doc-preview-tooltip`, что и KB-тултип (стили продублированы в `message.css`)

### Message — рендеринг сообщений

**Markdown-рендеринг (светлый стиль KB):**
- `ReactMarkdown` + `remarkGfm`

- Все ссылки проходят через `ChatDocLink`: внутренние `/?doc=N` → превью+навигация, внешние → обычная `<a target="_blank">`

**CodeBlock — копируемые блоки кода:**

**Tool Calls:**

### useKnowledgeBase — центральный хук

**Пропсы (из `useAppNavigation` через `KnowledgeBase.jsx`):**

| Проп | Тип | Описание |
|---|---|---|
| `docId` | `string \| null` | Какой документ показать |

| `onSearch(query, mode)` | callback | KB сообщает навигации: запущен поиск |
| `onTabChange(tab)` | callback | KB сообщает навигации: сменилась вкладка детали |

**Хук НЕ пишет URL и НЕ слушает popstate** — этим владеет `useAppNavigation`.

**Состояние:**
- `tree` — дерево узлов
- `selectedNode` — выбранный узел
- `activeTab` — вкладка (summary/content/contents/attachments)
- `searchQuery`, `searchMode`, `searchResults`
- `showAddModal`, `notFoundDocId`, `docLoadError`, `saveError`
- `refreshing`, `moveConfirm`, `discardConfirm`

**Загрузка дерева:**
- `loadTree()` — корневой уровень через `GET /api/documents/children`
- `handleLoadChildren(parentId, page)` — ленивая загрузка при раскрытии папки
- `refreshScope(parentId)` — после CRUD-операций

**Навигация (централизованная):**
- `selectNode(nodeOrId)` — выбор узла. Всегда делает `fetchById` для получения полного `description`. Для SearchResult (без поля `type`) — также `fetchById` перед отображением. Вызывает `onOpenDoc(id, 'summary')` с `notify: true`
- `navigateToDocById(docId, opts)` — загрузка цепочки ancestors → раскрытие → выбор. Используется при восстановлении из URL (с `notify: false`, чтобы избежать петли)
- `applySelectNode(node, {notify})` — `notify: true` для пользовательских действий (уведомляет навигацию), `notify: false` когда выбор пришёл ИЗ навигации (prop `docId`)
- **Больше нет:** `getUrlState`, `setKBUrlState`, `popstate`-слушателя, `app:navigate-doc`-слушателя, инициализации из URL

**Реакция на навигацию (props):**
- Единственный `useEffect` на `navDocId`, `navSearch`, `navDocTab`, `navMode`
- `navDocId` → `navigateToDocById(docId, {notify: false})` (не уведомляет навигацию обратно)
- `navSearch` → `performSearch(query, mode)`
- `lastNavDocRef` / `lastNavSearchRef` — защита от повторной обработки того же значения

**Защита от дублирующихся запросов:**
- `didInitRef` — предотвращает двойной вызов инициализации в StrictMode
- `inflightChildrenRef` — дедупликация параллельных запросов children (tree expand + folder detail)

**Защита от потери несохранённых изменений (editorDirtyStore):**

`useKnowledgeBase` оборачивает все навигационные действия в guard-функцию:

- `guard(action)` — если `isEditorDirty()`, откладывает `action` в `pendingActionRef` и показывает `discardConfirm`-модалку. Иначе выполняет сразу
- `handleDiscardConfirm` — сбрасывает dirty-флаги через `clearEditorDirty()`, выполняет отложенное действие
- `handleDiscardCancel` — сбрасывает отложенное действие, закрывает модалку

**Обёрнутые (guarded) методы:**
- `guardedSelectNode` — выбор узла (дерево, поиск, doc-ссылка)
- `guardedChangeTab` — смена вкладки детали
- `guardedSearch` — запуск поиска
- `guardedRefresh` — обновление документа (перезагружает с API → правки теряются)

Также хук слушает `beforeunload` — если редактор dirty, браузер покажет стандартное предупреждение при закрытии вкладки.

**CRUD:**
- Создание: `POST /api/documents`
- Обновление: `PUT /api/documents/{id}` (оптимистичное)
- Удаление: `DELETE /api/documents/{id}` с подтверждением
- Переименование: `handleRename(id, title)`

**AI-саммаризация:**
- `handleSummarize(id)` — вызывает `api.summarize(id)` → `POST /api/documents/{id}/summarize`
- Патчит `summary`, `summaryStale`, `summarySourceVersion` в `selectedNode` и `tree`

**Drag & Drop:**
- `handleReorder(dropInfo)` — оптимистичный UI → `PATCH /api/documents/reorder`
- При смене родителя — подтверждение через `MoveConfirmModal`
- Откат при ошибке

**Поиск:**
- `GET /api/documents/search?q=...&mode=...`
- Режимы: hybrid, semantic, keyword
- При активном поиске вызывает `onSearch(query, mode)` (уведомляет навигацию)

**Смена вкладки:**
- `changeTab(tab)` — локально + `onTabChange(tab)` (уведомляет навигацию для URL)

### editorDirtyStore — модуль-уровневое хранилище dirty-флагов

**Проблема:** `DocumentDetail` рендерится с `key={node.id}`, поэтому при смене документа редактор размонтируется — его внутренний `dirty`-стейт теряется до того, как навигация принимает решение.

**Решение:** модуль-уровневый `Set`-реестр (как кэш в `useDocPreview`):

| Функция | Назначение |
|---|---|
| `setEditorDirty(id, value)` | Зарегистрировать/снять dirty-флаг экземпляра редактора |
| `isEditorDirty()` | `true` если хотя бы один редактор dirty |
| `clearEditorDirty()` | Сбросить все флаги (после подтверждения discarding) |
| `subscribeEditorDirty(cb)` | Подписка на изменения (возвращает unsubscribe) |

**Ключевание по instance id** (а не один булев флаг): инлайн-редактор и полноэкранный «развернуть» могут быть смонтированы одновременно. Чистый экземпляр не должен затирать dirty-статус другого.

**Интеграция в MarkdownEditor:**
- При монтировании: `setEditorDirty(dirtyId, dirty)` в `useEffect` на `[dirty, previewOnly]`
- При размонтировании: cleanup `setEditorDirty(dirtyId, false)` — чтобы «мёртвый» редактор не блокировал навигацию
- `previewOnly`-экземпляры (SummarySection) никогда не регистрируются

### TreeNode — узел дерева

- **Раскрытие:** `toggleOpenVisual` вместо `toggleOpen` — не делает отдельный fetch, полагается на shared loader из `useFolderChildren`. Это устраняет дублирующиеся запросы при раскрытии папки.
- **«Load more»:** кнопка внизу списка детей, если `totalElements > children.length`
- **Drag & Drop:** module-level `dragState` (обход ограничений браузера)
  - Системные узлы не перетаскиваются
  - Drop-позиции: before, after, inside (для папок)
  - Визуальные индикаторы через CSS-классы
- **Авто-открытие:** `_openOnLoad` для навигации по прямой ссылке
- **Авто-раскрытие:** если выбранный узел — потомок

### useFolderChildren — shared loader

- Проверяет `treeCacheIsComplete` — если дети уже загружены, не делает повторный запрос
- Использует общий `loadChildren` вместо прямого `api.fetchChildren`
- `FolderDetail` передаёт `onLoadChildren` — загрузка детей папки через shared loader, а не отдельный запрос

### HistoryModal — история изменений описания

Полноэкранная панель (263 строки), открывающаяся по кнопке «История» из `DocumentDetail`, `FolderDetail` и тулбара `MarkdownEditor`. Позволяет просматривать все версии описания документа с diff-представлением и восстанавливать любую предыдущую версию.

**Загрузка данных:**
- `api.fetchHistory(id)` → `GET /api/documents/{id}/history`
- Ответ: массив записей `DocumentHistoryEntry` с полями `descriptionVersion`, `description`, `updatedAt`, `updatedBy`

**Структура панели (две колонки):**
- **Левая колонка** — список версий (хронологически, сверху новая). Каждая версия показывает дату, автора и сниппет описания. Текущая версия помечена бейджем «текущая». Активная версия подсвечена
- **Правая колонка** — предпросмотр выбранной версии (рендеринг Markdown через `ReactMarkdown` + `remarkGfm`)

**Diff-представление:**
- При выборе версии (не текущей) — кнопка «Сравнить с текущей»
- Diff вычисляется библиотекой `diff` (^9.0.0): `diff.diffWords(oldText, newText)`
- Добавленные фрагменты подсвечиваются зелёным фоном (`.history-diff-added`), удалённые — красным с зачёркиванием (`.history-diff-removed`)

**Восстановление версии:**
- Кнопка «Восстановить эту версию» (только для не-текущих версий)
- При клике: `PUT /api/documents/{id}` с `description` выбранной версии
- После успешного восстановления: модалка закрывается, документ обновляется (refresh)

**Закрытие:** Esc или клик по оверлею.

### AISummarySection — AI-саммаризация документа

Компонент внутри `DocumentDetail`, отображается на вкладке Summary над секцией «About». Три состояния:

| Состояние | UI |
|---|---|
| **Нет summary** (`node.summary === null`) | Заголовок «AI Summary», кнопка «Сгенерировать summary». Если нет description — кнопка заблокирована + подсказка «Сначала добавьте описание во вкладке Content» |

### MarkdownEditor

- **Тулбар:** H2, Bold, Italic, Code, Quote, List, Link, История

### DocLinkTooltip — ховер-превью для KB-ссылок

Обёртка для ссылок `/?doc=N` в рендеренном Markdown. При наведении показывает портал-тултип с превью документа:

- **Кэширование:** `useDocPreview` — сначала ищет в tree (мгновенно), затем в module-level cache, затем fetch
- **Дедупликация:** параллельные запросы к одному документу ждут один и тот же промис
- **Навигация:** через проп `onNavigate` (в KB это `selectNode`, принимающий id). `selectNode → fetchFullAndSelect(id, {notify:true})` → документ грузится, выбирается, и `useAppNavigation` обновляет URL. Фолбэк на `window.dispatchEvent('app:navigate-doc')` оставлен для обратной совместимости
- **Задержка:** 180ms до показа, 200ms до скрытия
- **Позиционирование:** фиксированное, авто-флип (сверху/снизу), не выходит за границы экрана
- **Содержимое тултипа:** заголовок + тип, AI-summary (если есть) или сниппет description, дата, кнопка «Открыть →»

### useDocPreview — кэш превью документов

Модуль-уровневый кэш (живёт всю жизнь страницы):

1. **Tree lookup:** если узел есть в дереве с загруженным description → мгновенный ответ
2. **Module cache:** `Map<id, node|'loading'|'error'>` — повторные ховеры на ту же ссылку мгновенны
3. **Inflight dedup:** если запрос уже в полёте → подписывается на результат через `listeners`
4. **Fresh fetch:** `api.fetchById(id)` → сохраняет в кэш → уведомляет подписчиков

### useAtMention — автокомплит @mention

Хук, подключаемый к textarea через ref:

- **Детект:** regex `(?:^|[\\s\\n])@([^\\s@]*)$` на тексте до курсора
- **Дебаунс:** 200ms перед запросом
- **Поиск:** `GET /api/documents/search-by-name?name=...&limit=10`
- **Позиционирование:** mirror-div для точного положения каретки
- **Клавиатура:** ↑↓ — навигация, Enter — выбрать, Esc — закрыть
- **Вставка:** заменяет `@query` на `[Название](/?doc=ID)`, курсор после ссылки

### Поиск (SearchResults)

- Карточки результатов: заголовок, дата, хлебные крошки, сниппет
- Хлебные крошки приходят с бэкенда в `parentList` (батчевый CTE-запрос), а не строятся на фронте через `findPath()`
- Заголовок — `<a href>` с поддержкой средней кнопки мыши и Ctrl+клик
