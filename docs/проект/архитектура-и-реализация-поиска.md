## 1. Обзор

Поиск в Knowledge Base — трёхрежимная система: **keyword**, **semantic** и **hybrid**. Все три режима доступны через единый эндпоинт `GET /api/search` с параметром `mode`.

| Режим | Механизм | Когда применять |
|---|---|---|
| **Keyword** | SQL `ILIKE` по `title`, `summary`, `description` | Точные слова известны |
| **Semantic** | Эмбеддинги → pgvector `cosine similarity` | Поиск по смыслу |
| **Hybrid** (по умолчанию) | Параллельный keyword + semantic с объединением | Универсальный |

## 3. Точка входа: `DocumentController`

Файл: `backend/.../controller/DocumentController.java` (233 строки)

Единый эндпоинт `GET /api/search` — простой switch по режиму:

```java
return switch (mode.toLowerCase()) {
    case "semantic" -> service.semanticSearch(q, threshold, limit);
    case "hybrid"   -> service.hybridSearch(q, threshold, limit, kwWeight, semWeight);
    default         -> service.search(q);  // keyword
};
```

**Параметры запроса:**

| Параметр | Тип | По умолчанию | Назначение |
|---|---|---|---|
| `q` | String | — | Поисковый запрос |
| `mode` | String | `keyword` | `keyword` / `semantic` / `hybrid` |
| `threshold` | Double? | `null` | Порог схожести для semantic/hybrid |
| `limit` | Integer? | `null` | Макс. количество результатов |
| `kwWeight` | Double? | `null` | Вес keyword (только hybrid) |
| `semWeight` | Double? | `null` | Вес semantic (только hybrid) |

Также здесь:
- `GET /api/documents/search-by-name` — автокомплит @mention (поиск по фрагменту названия)
- `POST /api/documents/reindex` — полная переиндексация всех документов и вложений

## 4. Бизнес-логика: `DocumentService`

Файл: `backend/.../service/DocumentService.java` (760 строк)

### 4.1 Keyword-поиск (`search`, строки 567–578)

```java
public List<SearchResult> search(String q) {
    return repo.search(q).stream()
            .map(e -> new SearchResult(
                    String.valueOf(e.getId()),
                    e.getTitle(),
                    generateSnippet(e.getDescription(), q.toLowerCase()),
                    e.getUpdatedAt(),
                    e.getSummary()))
            .collect(Collectors.toList());
}
```

- Делегирует в `DocumentRepository.search()` — SQL `ILIKE` по `title`, `summary`, `description`
- Генерирует сниппет через `generateSnippet()` — фрагмент текста вокруг найденного слова
- Результаты сортируются по `updated_at DESC`, лимит 20

### 4.2 Semantic-поиск (`semanticSearch`, строки 590–604)

```java
public List<SearchResult> semanticSearch(String q, Double threshold, Integer limit) {
    double t = threshold != null ? threshold : searchConfig.semantic().threshold();
    int l = limit != null ? limit : searchConfig.semantic().limit();
    return semanticSearchService.search(q, t, l).stream()
            .map(r -> new SearchResult(...))
            .collect(Collectors.toList());
}
```

- Делегирует в `SemanticSearchService.search()`
- Маппит `SemanticSearchResult` → `SearchResult` (единый DTO)

### 4.3 Hybrid-поиск (`hybridSearch`, строки 626–681)

Алгоритм объединения:

1. **Keyword hits** — нормализованный score: `(размер_списка − позиция) / размер_списка`
2. **Semantic hits** — score = cosine similarity (уже в диапазоне 0..1)
3. **Merge**: `hybridScore = kw × keywordWeight + sem × semanticWeight`
4. Сортировка по убыванию, top-N

```java
// Нормализация keyword-результатов
for (int i = 0; i < kwSize; i++) {
    kwScores.put(kwResults.get(i).getId(), (double) (kwSize - i) / kwSize);
}

// Объединение score'ов
double score = kw * kwScores.getOrDefault(id, 0.0)
             + sem * semScores.getOrDefault(id, 0.0);
```

Дубликаты (один документ, найденный обоими методами) схлопываются — сниппет берётся из keyword-результата, если доступен.

## 5. Семантический движок: `SemanticSearchService`

Файл: `backend/.../service/SemanticSearchService.java` (299 строк)

### Индексация

| Метод | Назначение |
|---|---|
| `indexDocument(id, title, description)` | Upsert эмбеддинга одного документа |
| `reindexAll()` | Полная переиндексация документов + вложений, батчами |
| `deleteIndex(id)` | Удаление эмбеддинга документа |

`reindexAll()` работает в двух фазах:
1. **Документы** — все строки из `documents`, батчами по `reindexBatchSize`
2. **Вложения** — все строки из `attachments`, с усечением контента до ~6000 символов

### Поиск

| Метод | Область поиска |
|---|---|
| `search(query, threshold, limit)` | Только документы (backward compat) |
| `searchAll(query, threshold, limit)` | Документы + вложения, объединённые по similarity |

`searchAll` выполняет два независимых векторных запроса (к `document_embeddings` и `attachment_embeddings`), затем сливает и сортирует результаты по убыванию similarity.

## 6. Эмбеддинги: `EmbeddingService`

Файл: `backend/.../service/EmbeddingService.java` (308 строк)

Три слоя оптимизации:

| Слой | Механизм |
|---|---|
| **Postgres-кэш** | SHA-256(text) + model → `embedding_cache`. Повторный текст не ходит в OpenAI. `last_used_at` обновляется при каждом cache hit для LRU-вытеснения |
| **Батчинг** | Все cache-miss тексты отправляются одним HTTP-запросом к OpenAI |
| **Чанкинг** | Длинные тексты → `TextChunker` → mean pooling векторов |

### Ключевые методы

```java
// Одиночный эмбеддинг (для поискового запроса)
EmbeddingResponse embed(String text)

// Батчевый эмбеддинг (для reindex)
List<float[]> embedBatch(List<String> texts)

// Эмбеддинг документа: title + "\n" + description
EmbeddingResponse embedDocument(String title, String description)
```

### Алгоритм `embedBatch`

1. **Cache hits** — обслуживаются из Postgres без API-вызова
2. **Короткие cache-miss** — отправляются одним батчевым запросом к OpenAI
3. **Длинные тексты** (не помещаются в один чанк) — чанкуются и эмбедятся индивидуально, затем mean pooling

### Mean Pooling

```java
static float[] meanPool(List<float[]> vectors) {
    int dim = vectors.get(0).length;
    float[] sum = new float[dim];
    for (float[] v : vectors) {
        for (int i = 0; i < dim; i++) sum[i] += v[i];
    }
    float n = vectors.size();
    for (int i = 0; i < dim; i++) sum[i] /= n;
    return sum;
}
```

## 7. Чанкер: `TextChunker`

Файл: `backend/.../service/TextChunker.java` (190 строк)

Каскадная стратегия разбиения:

1. **Параграфы** (`\n\n`) — предпочтительная единица
2. **Предложения** (`[.!?]`) — если параграф слишком длинный
3. **Слова** — последнее средство для очень длинных предложений

**Параметры:**
- `maxTokens = 512` (≈2048 символов, при 4 chars/token)
- `overlapTokens = 64` — перекрытие между соседними чанками

**Оценка токенов:** 1 токен ≈ 4 символа (консервативно для латиницы; CJK-текст будет недооценён, но безопасен для соблюдения жёсткого лимита).

Перекрытие (`overlapTokens`) гарантирует, что запрос, попадающий на границу двух чанков, всё равно найдёт релевантный документ.

## 8. Репозитории и SQL

### 8.1 `DocumentRepository` — keyword-поиск

Файл: `backend/.../repository/DocumentRepository.java` (124 строки)

```sql
SELECT * FROM documents
WHERE (title ILIKE '%' || :q || '%'
       OR summary ILIKE '%' || :q || '%'
       OR description ILIKE '%' || :q || '%')
ORDER BY updated_at DESC
LIMIT 20
```

- Регистронезависимый поиск (`ILIKE`)
- Ищет каждое слово запроса отдельно
- Сортировка: сначала свежие документы

### 8.2 `DocumentEmbeddingRepository` — semantic-поиск

Файл: `backend/.../repository/DocumentEmbeddingRepository.java` (99 строк)

Использует raw `JdbcTemplate`, потому что Spring Data JDBC не поддерживает оператор `<=>` (cosine distance) нативно.

```sql
SELECT
    d.id          AS document_id,
    d.title,
    d.description,
    d.updated_at,
    d.summary,
    1 - (de.embedding <=> ?::vector) AS similarity
FROM document_embeddings de
JOIN documents d ON d.id = de.document_id
WHERE 1 - (de.embedding <=> ?::vector) >= ?
ORDER BY similarity DESC
LIMIT ?
```

- `<=>` — оператор **cosine distance** из pgvector
- `similarity = 1 − distance`
- Вектор передаётся как строковый литерал: `[0.12,0.34,...]`

### 8.3 `AttachmentEmbeddingRepository`

Файл: `backend/.../repository/AttachmentEmbeddingRepository.java` (94 строки)

Зеркальная структура, но джойнится к `attachments` вместо `documents`. Используется в `SemanticSearchService.searchAll()`.

## 10. Модели данных поиска

### 10.1 `SearchResult` — единый DTO

Используется всеми тремя режимами. Поля: `id`, `title`, `snippet`, `updatedAt`, `summary`.

### 10.2 `SemanticSearchResult` — результат векторного поиска

Расширяет `SearchResult` полем `similarity` (double, 0..1).

### 10.3 `DocumentEmbeddingEntity`

Таблица `document_embeddings`. Поля: `document_id` (FK → documents), `embedding` (pgvector `vector`), `model`, `updated_at`.

### 10.4 `EmbeddingCacheEntity`

Таблица `embedding_cache`. Ключ: `(textHash, model)`. Поля: `embedding` (pgvector), `created_at`, `last_used_at`. LRU-вытеснение по `last_used_at`, TTL 30 дней. Подробнее — [Embedding-кэш](модели-данных/embedding-кэш.md).

## 12. Жизненный цикл эмбеддинга

```
Создание/обновление документа
        │
        ▼
DocumentService.update()
        │
        ▼
SemanticSearchService.indexDocument(id, title, description)
        │
        ▼
EmbeddingService.embedDocument(title, description)
        │
        ├─ Текст помещается в один чанк?
        │   ├─ Да → lookupCache → hit? → вернуть из кэша
        │   │                      → miss? → callApi → writeCache → вернуть
        │   └─ Нет → TextChunker.split()
        │           → для каждого чанка: lookupCache → (hit | callApi + writeCache)
        │           → meanPool() → вернуть
        │
        ▼
DocumentEmbeddingRepository.save(entity)
```
