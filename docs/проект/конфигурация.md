### Spring AI — OpenAI
```yaml
spring:

      base-url: ${AI_BASE_URL}
      api-key: ${AI_API_KEY}
      chat:
        options:
          model: ${AI_MODEL}
          max-tokens: 30000
          temperature: 0.1
          stream-usage: true
          extra-body:
            thinking:
              type: disabled
      embedding:
        base-url: ${AI_EMBED_BASE_URL:${AI_BASE_URL}}
        api-key: ${AI_EMBED_API_KEY:${AI_API_KEY}}
        embeddings-path: /v1/embeddings
        options:
          model: ${kb.embedding.model:bge-m3}
    chat:
      memory:
        repository:
          jdbc:
            initialize-schema: never
      client:
        enabled: off
    retry:
      max-attempts: 2
```

### База данных
```yaml
spring:
  datasource:
    driver-class-name: org.postgresql.Driver
    url: ${DATASOURCE_URL:jdbc:postgresql://localhost:5432/knowledgebase}
    username: ${DATASOURCE_USERNAME:knowledgebase}
    password: ${DATASOURCE_PASSWORD:knowledgebase}
  flyway:
    enabled: true
    locations: classpath:db/migration
    validate-on-migrate: false
```

### Эмбеддинги и чанкинг
```yaml
kb:
  embedding:
    model: ${AI_EMBED_MODEL}
    reindex-batch-size: 50
    cache:
      enabled: true
      ttl-days: 30
      cleanup-cron: "0 0 2 * * *"
    chunker:
      max-tokens: 512
      overlap-tokens: 64
```

### Поиск
```yaml
kb:
  search:
    semantic:
      enabled: true
    keyword:
      limit: 20
    semantic:
      threshold: 0.20
      limit: 20
    hybrid:
      keyword-weight: 0.4
      semantic-weight: 0.6
      threshold: 0.20
      limit: 20
```

> **`semantic.enabled: true`** — включает/отключает семантический поиск. При `false` `EmbeddingService.embed()` возвращает пустой результат, а `SemanticSearchService` пропускает все операции (index, search, reindex). Используется в H2-профиле, где нет pgvector.

### Atlassian (Jira + Confluence)

kb:

spring:

## Профиль H2 (`application-h2.yaml`)

Альтернативный профиль для запуска без PostgreSQL и без AI-ключа. Активируется через `SPRING_PROFILES_ACTIVE=h2`.

```yaml
spring:
  datasource:
    driver-class-name: org.h2.Driver
    url: jdbc:h2:file:./local-db/h2;MODE=PostgreSQL;DATABASE_TO_LOWER=TRUE
    username: sa
    password:
  flyway:
    locations: classpath:db/migration-h2
  sql:
    init:
      mode: always

kb:
  search:
    semantic:
      enabled: false
```

| Параметр | Значение |
|---|---|
| **Драйвер** | `org.h2.Driver` |
| **URL** | `jdbc:h2:file:./local-db/h2;MODE=PostgreSQL;DATABASE_TO_LOWER=TRUE` |
| **Режим** | PostgreSQL-совместимый |
| **Хранение** | Файловое (`./local-db/h2.mv.db`) |
| **Миграции** | `db/migration-h2/` (отдельный набор, адаптированный под H2) |
| **Семантический поиск** | Отключён (`semantic.enabled: false`) |
| **H2-консоль** | Доступна на `:8080/h2-console` |

### H2JdbcConfig

При профиле `h2` активируется `H2JdbcConfig` — конфигурация Spring Data JDBC с диалектом `JdbcPostgresDialect` для корректной работы H2 в PostgreSQL-режиме.

## Docker Compose H2 (`docker/docker-compose-h2.yaml`)

Облегчённый вариант без PostgreSQL — только backend с H2:

```yaml
services:
  backend:
    build:
      context: ..
      dockerfile: docker/Dockerfile
    container_name: kb-backend
    env_file:
      - ./example.env
    environment:
      - SPRING_PROFILES_ACTIVE=h2
    ports:
      - "8080:8080"
    volumes:
      - kb-h2-data:/app/local-db
      - ${PROJECT_PATH_MOUNT:-../}:/project:ro
      - ${DOCUMENTS_EXPORT_PATH:-./export}:/export
    healthcheck:
      test: ["CMD-SHELL", "curl -f http://localhost:8080/actuator/health || exit 1"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s

volumes:
  kb-h2-data:
```

> **Запуск:** `docker compose -f docker/docker-compose-h2.yaml up -d`. Не требует PostgreSQL и AI-ключа. Данные H2 хранятся в volume `kb-h2-data`.

## Docker Compose (`docker/docker-compose.yaml`)

Два сервиса: **PostgreSQL 17 + pgvector** и **backend** (Spring Boot). Хранение файлов через вложения в PostgreSQL.

```yaml
services:
  postgres:
    image: pgvector/pgvector:pg17
    container_name: kb-postgres
    environment:
      POSTGRES_DB: knowledgebase
      POSTGRES_USER: knowledgebase
      POSTGRES_PASSWORD: knowledgebase
    ports:
      - "5432:5432"
    volumes:
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U knowledgebase"]
      interval: 5s
      timeout: 5s
      retries: 5

  backend:
    build:
      context: ..
      dockerfile: docker/Dockerfile
    container_name: kb-backend
    depends_on:
      postgres:
        condition: service_healthy
    env_file:
      - ./example.env
    ports:
      - "8080:8080"
    volumes:
      - ${PROJECT_PATH_MOUNT:-../}:/project:ro
      - ${DOCUMENTS_EXPORT_PATH:-./export}:/export
    healthcheck:
      test: ["CMD-SHELL", "curl -f http://localhost:8080/actuator/health || exit 1"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s

volumes:
  pgdata:
```

## Dockerfile (`docker/Dockerfile`)

Трёхстадийная сборка:

1. **Build stage** (`eclipse-temurin:25-jdk-alpine`) — сборка bootJar с кэшированием Gradle/Node/Yarn зависимостей, `git config --global --add safe.directory /build`
2. **Extract stage** (`eclipse-temurin:25-jre-alpine`) — распаковка слоёв JAR: `java -Djarmode=tools -jar app.jar extract --layers --destination extracted`
3. **Runtime stage** (`eclipse-temurin:25-jre-jammy`) — минимальный образ с Git и curl (`apt-get install`), `groupadd/useradd` для пользователя `kb`, копирование слоёв из extract-стадии (dependencies, spring-boot-loader, snapshot-dependencies, application), создание `/app/local-db` с правами `kb`, entrypoint `start.sh`

## Миграции Flyway

Миграции находятся в `backend/src/main/resources/db/migration/`. Flyway применяет их автоматически при старте приложения.

| Миграция | Описание |
|---|---|
| `V2026.05.24__document_history.sql` | Добавляет колонку `version` в `documents` (optimistic locking) и таблицу `document_history` (снапшоты перед каждым обновлением) |
| `V2026.05.25__document_summary.sql` | Добавляет колонки `summary`, `summarySourceVersion`, `descriptionVersion` в `documents` и `document_history` |
| `V2026.05.26__fix_description_index.sql` | Замена GiST-индекса на GIN для `description` — исправляет ошибку «index row requires 8360 bytes» на больших документах (триграмное представление GiST превышало лимит btree-страницы 8191 байт) |

### H2-миграции (`db/migration-h2/`)

Отдельный набор миграций для профиля `h2`. Используются при `SPRING_PROFILES_ACTIVE=h2`.

| Миграция | Описание |
|---|---|
| `V1__init.sql` | Полная схема БД для H2 (145 строк) — все таблицы, индексы, constraints, адаптированные под H2 в PostgreSQL-режиме |

## resolveAndLockAll

Задача `resolveAndLockAll` в корневом `build.gradle` разрешает все конфигурации для генерации lock-файлов. Требует явного указания `--write-locks` и несовместима с configuration cache:

```bash
./gradlew resolveAndLockAll --write-locks
```

## Tree-sitter и платформозависимость

Tree-sitter — нативная библиотека для структурного анализа кода (используется в `OutlineService`). Пять зависимостей: ядро + грамматики для Java, JavaScript, TypeScript, Python.

- **Платформозависимость:** нативная библиотека грузится под конкретную ОС/архитектуру. Сборочный и рантайм-образ должны совпадать (linux-x64, mac-arm64 и т.д.), иначе `UnsatisfiedLinkError`.
- **Фолбэк на regex:** конструктор `TreeSitterOutlineParser` ловит `Throwable`, ставит `available=false`, и `OutlineService` переключается на regex-парсер. Это валидный режим работы.
- **Потокобезопасность:** `TSParser` не потокобезопасен — создаётся новый экземпляр на каждый вызов. `TSLanguage` иммутабелен и кэшируется.
- **Опциональность:** если tree-sitter не нужен, можно не добавлять зависимости — outline будет работать через regex.
