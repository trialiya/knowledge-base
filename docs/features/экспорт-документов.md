# Экспорт документов

Экспорт базы знаний в файловую систему через API `POST /api/documents/admin/export` и через админ-панель. Позволяет выгрузить всю структуру документов в читаемом формате — Markdown + опционально YAML-метаданные.

> **Связанные документы:** [API Reference](../проект/api-reference.md) · [Конфигурация](../проект/конфигурация.md) · [Документы](../проект/модели-данных/документы.md)

## Формат экспорта

Экспорт создаёт следующую структуру в директории, указанной в `kb.documents.export-path`:

```
export-dir/
├── .index.md              # упорядоченный список всех корневых элементов
├── vvedenie.md             # документ → .md файл (только description, без заголовка)
├── vvedenie.yaml           # метаданные документа (только при meta=true)
├── arkhitektura.md
├── arkhitektura.yaml
├── modeli-dannykh/         # папка (имя: lower-case, спецсимволы → дефисы)
│   ├── .index.md           # упорядоченный список детей папки
│   ├── .content.md         # описание папки (всегда создаётся, может быть пустым)
│   ├── .meta.yaml          # метаданные папки (только при meta=true)
│   ├── dokumenty.md
│   ├── dokumenty.yaml
│   ├── chat.md
│   └── ...
└── features/
    ├── .index.md
    ├── .content.md
    └── ...
```

### Типы файлов

| Файл | Назначение |
|---|---|
| `название.md` | Тело документа — только description, без заголовка (заголовок в метаданных) |
| `название.yaml` | Метаданные документа: id, title, type, parentId, position, updatedAt (только при `meta=true`) |
| `.index.md` | Упорядоченный список детей папки в Markdown; папки ссылаются на `.content.md`, документы — на `.md` |
| `.content.md` | Описание папки (создаётся всегда, даже для пустых папок) |
| `.meta.yaml` | Метаданные папки (только при `meta=true`) |

### Правила именования

- Заголовки документов приводятся к нижнему регистру, спецсимволы и пробелы заменяются на дефисы: `safeName()` → `modeli-dannykh`, `vvedenie`
- Кириллица сохраняется как есть (транслитерация не применяется)
- При совпадении имён и `replace=false` добавляется суффикс: `doc.md` → `doc-1.md` → `doc-2.md`
- При `replace=true` существующие файлы перезаписываются
- Служебные файлы папок используют префикс `.` (скрытые)

## Перезапись ссылок

Все внутренние ссылки вида `[Название](/?doc=ID)` в содержимом документов автоматически заменяются на относительные пути файловой системы:

```
[Архитектура](../проект/архитектура.md)  →  [Архитектура](arkhitektura.md)
[Модели данных](../проект/модели-данных/.content.md) →  [Модели данных](modeli-dannykh/.content.md)
```

Ссылки на папки резолвятся в `.content.md`. Если целевой документ не найден в дереве — ссылка остаётся без изменений. Это делает экспортированные документы полностью автономными — ссылки работают при просмотре в любом Markdown-редакторе или на GitHub.

## API

### `POST /api/documents/admin/export`

Экспортирует всю базу знаний в файловую систему.

**Параметры запроса:**

| Параметр | Тип | По умолчанию | Описание |
|---|---|---|---|
| `meta` | boolean | `true` | Создавать ли `.yaml` и `.meta.yaml` файлы с метаданными |

**Поведение:**

- Экспорт всегда полный — все документы и папки выгружаются заново
- Ответ: `204 No Content` (без тела)
- Количество записанных файлов логируется на сервере

**Ошибки:**

| Код | Причина |
|---|---|
| `500` | Ошибка файловой системы (нет прав, нет места и т.д.) |

### `POST /api/documents/admin/reindex`

Запускает полную переиндексацию семантического поиска — пересчитывает embedding-векторы для всех документов.

**Ответ:** `200 OK` с телом `{ "indexed": <количество> }`.

## Админ-панель (UI)

Экспорт доступен через админ-панель: раздел **Администрирование → Массовые операции**.

Компонент `BulkOperations.jsx` предоставляет:

- **Экспорт всех документов** — кнопка «Выгрузить», чекбокс «С метаданными (.yaml)», индикация состояний (idle/running/done/error)
- **Импорт документов** — кнопка «Загрузить» (заглушка, бэкенд пока не реализован)

При успешном экспорте показывается зелёный бейдж «готово» с сообщением. При ошибке — красный бейдж «ошибка» с подсказкой проверить `DOCUMENTS_EXPORT_PATH`.

## Конфигурация

В `application.yaml`:

```yaml
kb:
  documents:
    export-path: ${KB_DOCUMENTS_EXPORT_PATH:./exported-docs}
    replace: true   # перезаписывать существующие файлы (false → суффиксы -1, -2...)
```

| Параметр | Переменная окружения | Описание |
|---|---|---|
| `kb.documents.export-path` | `KB_DOCUMENTS_EXPORT_PATH` | Путь к директории для экспорта |
| `kb.documents.replace` | — | Перезаписывать ли существующие файлы при совпадении имён |

## Ключевые файлы

| Файл | Назначение |
|---|---|
| `backend/.../service/DocumentExportService.java` | Двухпроходный экспорт: сбор id→path + запись файлов с перезаписью ссылок |
| `backend/.../controller/DocumentController.java` | Эндпоинты `POST /admin/export` и `POST /admin/reindex` |
| `backend/.../config/model/DocumentsConfiguration.java` | Конфиг `export-path` и `replace` |
| `frontend/.../adminPanel/BulkOperations.jsx` | UI экспорта в админ-панели |
| `frontend/.../knowledgeBasePanel/api.js` | Методы `exportToFolder()` и `reindex()` |

## Примечания

- Вложения в экспорт не включаются
- История изменений (document_history) не экспортируется
- Формат оптимизирован для чтения человеком и хранения в Git
- `DocumentExportService` делает два прохода: первый собирает карту `id → абсолютный путь`, второй пишет файлы с перезаписью ссылок
