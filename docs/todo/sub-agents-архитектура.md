# Sub-agents в чате — проработка архитектуры

Статус: проектная проработка (код не менялся). Отправная точка — существующий
поисковый сабагент `SearchAgentService` (`searchCodebase`) и фоновый прогон
чата `ChatRunService`.

## 1. Что уже есть и что из этого переиспользуем

Сегодня в системе два «кирпича», из которых собирается всё остальное:

1. **Agent-as-tool (синхронный).** `SearchAgentService` — изолированный
   мини-агент: свой system prompt, ограниченный read-only набор инструментов,
   ручной tool-цикл через `ToolCallingManager` с жёстким капом итераций,
   деградация в summarize при ошибке/капе, учёт токенов. Родительская модель
   вызывает его как обычный tool (`SearchAgentFunction`) и *ждёт* результат
   внутри своего tool-цикла.
2. **Фоновый прогон + событийная шина.** `ChatRunService` уже умеет главное
   для асинхронности: запуск работы вне HTTP-запроса (`chatRunExecutor`),
   реестр активных прогонов, остановку через `Disposable`, и трансляцию хода
   в `ChatEventService`/`ConversationHub` (SSE с replay, переживает reload,
   виден всем вкладкам).

Вывод: **новую инфраструктуру событий и стриминга строить не нужно** — для
асинхронных сабагентов достаточно обобщить паттерн `ChatRunService` и добавить
персистентное состояние задач (сейчас `runs` живут только в памяти, потому что
результат прогона — это сами сообщения чата; у сабагента результат свой).

## 2. Два режима запуска сабагентов

### Режим A — agent-as-tool (синхронный, уже есть)

Модель сама решает, что ей нужен помощник, и блокируется на нём. Подходит для
задач, результат которых нужен *для продолжения текущего ответа*: поиск,
уточнение факта, извлечение структуры. `searchCodebase` — этот режим.

Что доработать: вынести из `SearchAgentService` универсальный «раннер» (см.
§4.1), чтобы новые tool-агенты (например `analyzeDocument`, `traceScenario`)
не копировали цикл/кап/summarize/учёт токенов.

### Режим B — фоновая задача (асинхронный, новый)

Работа, которая не нужна модели для текущего ответа, а нужна *пользователю
позже*: ревью диалога, аудит документации, генерация тест-плана. Запускается:

- пользователем — кнопка/команда в UI (`POST /api/chats/{id}/agents`);
- моделью — tool `startBackgroundAgent(type, task)`, который **сразу**
  возвращает `taskId` и не блокирует tool-цикл;
- системой — по событию (например, авто-ревью после `RUN_DONE` длинного
  диалога), опционально.

Ключевое отличие от режима A: у задачи есть жизненный цикл, состояние
переживает рестарт, результат доставляется отдельно от ответа модели.

## 3. Каталог идей (что запускать)

| Агент | Режим | Вход | Результат |
|---|---|---|---|
| `searchCodebase` (есть) | A | задача поиска | отчёт с path:line |
| **Ревью диалога** | B | история чата (readonly через `ChatMemoryService`) | отчёт: упущенные вопросы, противоречия, непроверенные утверждения модели, предложения следующих шагов |
| **Аудит документации** | B | документ(ы) БЗ + код | расхождения doc↔код со ссылками |
| **Тест-инженер** | B | сценарий/фича | тест-план или скелеты тестов (как документ БЗ) |
| **Суммаризация документа/ветки** | B | поддерево БЗ | сводный документ |
| **Fact-checker ответа** | B (авто) | последний ответ ассистента | список утверждений с подтверждением/опровержением из кода |
| **Трассировка сценария** | A | «как работает X целиком» | последовательность вызовов с местами в коде |

Все они ложатся в одну и ту же механику: prompt + ограниченный набор
инструментов + бюджет. Различие только в конфигурации — поэтому агент должен
описываться *декларативно* (см. `AgentDefinition` ниже), а не отдельным
сервисом на каждый тип.

## 4. Целевая архитектура

### 4.1. Ядро: `SubAgentRunner` (обобщение SearchAgentService)

```
record AgentDefinition(
    String type,            // "search" | "dialog-review" | "doc-audit" | ...
    String modelId,
    Resource systemPrompt,
    List<String> allowedTools,   // recursion guard: никогда не содержит агентские tools
    int maxIterations,
    int maxTokens)
```

`SubAgentRunner.run(AgentDefinition, task, toolContext, progressListener)` —
весь текущий цикл `SearchAgentService` (кап, summarize с `tool_choice=none`
ради кэша, self-correcting ошибки, `TokenUsage`) переезжает сюда без
изменений логики. `SearchAgentService` становится тонкой конфигурацией поверх
раннера. Новый параметр `progressListener` (колбэк «шаг N, вызван tool X») —
для live-статуса асинхронных задач; в синхронном режиме — no-op.

`SubAgentConfig` обобщается в map `kb.chat.sub-agents.<type>.*` в
`application.yaml` — новый агент добавляется prompt-файлом + секцией конфига.

### 4.2. Асинхронный слой: `AgentTaskService`

По образу `ChatRunService`, но с персистентным состоянием.

**Таблица `agent_task`** (миграции в `db/migration` и `db/migration-h2`,
плюс `db/sample-data.sql` + `SampleDataFixtureTest`):

```
id UUID PK
conversation_id  -- к какому чату привязан (nullable для «глобальных» задач)
type             -- тип агента
status           -- PENDING | RUNNING | DONE | FAILED | CANCELLED
task_text        -- вход
result_text      -- отчёт (заполняется в DONE/частично в FAILED)
progress         -- JSON: {hops, lastTool, note} — для UI
usage            -- JSON: {prompt, completion, total}
created_by, created_at, started_at, finished_at
result_message_id -- id сообщения чата с результатом, если доставлен (см. §5)
```

БД — источник истины о состоянии; SSE — только live-доставка. Это решает
и восстановление после рестарта: при старте приложения все `RUNNING` задачи
помечаются `FAILED("interrupted by restart")` (перезапуск — вручную; резюмить
tool-цикл с середины не пытаемся, это несоразмерно сложно).

**Исполнение:** отдельный ограниченный пул `agentTaskExecutor` (не
`chatRunExecutor` — фоновое ревью не должно отбирать поток у генерации
ответа), лимит одновременных задач на пользователя и глобально. Остановка —
кооперативная: `AgentTaskService.cancel(taskId)` ставит флаг, `SubAgentRunner`
проверяет его на границе каждой итерации (у нас нет стрима, который можно
`dispose()`, — цикл синхронный).

**События:** новые `ChatEventType`: `AGENT_STARTED`, `AGENT_PROGRESS`,
`AGENT_DONE`, `AGENT_FAILED`, `AGENT_CANCELLED` — публикуются в хаб
соответствующего `conversationId` через уже существующий
`ChatEventService.publishIfPresent` (если чат никто не смотрит — не страшно,
состояние есть в БД, вкладка при открытии заберёт его по REST). Важно: у хаба
есть логика выгрузки по `endRun` — агентские события должны идти *вне*
механики run-заявок (`activeByConversation` не трогаем: фоновая задача не
должна блокировать запуск обычной генерации в чате).

**REST:**

```
POST   /api/chats/{cid}/agents          {type, task} → {taskId}
GET    /api/chats/{cid}/agents          — список задач чата (+ активные)
GET    /api/agents/{taskId}             — статус/прогресс/результат
DELETE /api/agents/{taskId}             — cancel
```

### 4.3. Запуск моделью: tool `startBackgroundAgent`

Тонкий tool поверх `AgentTaskService.start(...)`: валидирует `type` по
allow-list, возвращает `taskId` и человекочитаемое «задача запущена, результат
появится в чате». Recursion guard тот же, что у `searchCodebase`: агентские
tools отсутствуют в `allowedTools` любых `AgentDefinition`. Плюс лимит: не
больше N активных фоновых задач на чат — иначе tool возвращает отказ текстом
(не исключением), модель сообщит пользователю.

## 5. Доставка результата

Три варианта, рекомендую первый:

1. **Результат = сообщение чата (рекомендуется).** По завершении
   `AgentTaskService` пишет отчёт в историю как ASSISTANT-сообщение с
   пометкой в `meta` (`agentTaskId`, `agentType`) — фронт рендерит его особой
   карточкой («Ревью диалога готово ▸»). Плюсы: результат виден после reload
   без нового UI-хранилища, попадает в контекст модели (она может обсуждать
   отчёт), работает существующий пайплайн persist/replay. Минус: длинный отчёт
   раздувает контекст → в историю класть *сжатую выжимку + taskId*, полный
   текст держать в `agent_task.result_text` и отдавать по
   `GET /api/agents/{taskId}` (тот же паттерн, что `tool_data` vs `meta` и
   `findToolCallDetail`).
   Внимание к гонке: писать в историю чата можно только между прогонами;
   если в момент завершения задачи идёт генерация (`activeRun` есть) —
   отложить вставку до `RUN_DONE`/`RUN_STOPPED` (подписаться на событие или
   проверять в `cleanup`).
2. Отдельная панель «Задачи» в UI без записи в историю — чище для контекста,
   но результат не обсуждаем с моделью и нужен новый UI-раздел.
3. Результат как документ БЗ (для тест-планов/суммаризаций уместно как
   *дополнительная* опция `saveAsDocument: true`).

## 6. Ограничители (обязательные с первого дня)

- Кап итераций и `maxTokens` — уже есть в раннере; добавить **бюджет токенов
  на задачу** (суммарный, из `TokenUsage`) с той же деградацией в summarize.
- Лимит параллельных задач (глобально/на пользователя/на чат).
- Таймаут задачи (wall-clock) — страховка от зависшего HTTP к провайдеру.
- Recursion guard: агентские tools никогда не входят в `allowedTools`.
- Все фоновые агенты — **read-only** инструменты (как у поискового);
  пишущие действия (создать документ) — только явные, отдельным шагом после
  подтверждения пользователем.

## 7. План внедрения (инкрементально)

1. **Рефакторинг без нового поведения:** извлечь `SubAgentRunner` +
   `AgentDefinition` из `SearchAgentService`; `searchCodebase` работает как
   раньше. Тесты на раннер (кап, summarize, cancel-флаг).
2. **Инфраструктура задач:** `agent_task` (обе миграции + fixture),
   `AgentTaskService`, executor, REST, `AGENT_*` события, восстановление
   после рестарта.
3. **Первый асинхронный агент — «Ревью диалога»:** prompt + read-only tool
   чтения истории чата; запуск кнопкой из UI; доставка выжимкой в историю
   (вариант 1 из §5). Это самый дешёвый агент для обкатки всей цепочки:
   вход у него уже в БД, кодовые инструменты не нужны.
4. **Tool `startBackgroundAgent`** + второй агент (аудит документации или
   тест-инженер) — по обратной связи.
5. Дальше по каталогу §3; авто-триггеры (fact-checker после `RUN_DONE`) —
   только после того, как ручной запуск докажет полезность: авто-агенты
   умножают расход токенов молча.
